第38课：我们现在离“跑第一款真正的 NES ROM”还差什么

这节课不讲新硬件细节，而是做一次阶段盘点。
目标是把你脑子里的问题从：

- “我又实现了一个模块”

变成：

- “我离第一款 ROM 运行还差哪几块关键拼图”

先给核心结论：

- 你现在已经过了“只能写 CPU 指令”的阶段
- 但还没到“随便丢 ROM 就能跑”的阶段
- 中间还差的是几块必须闭环的系统能力

---------

1. 先明确什么叫“跑第一款 ROM”

这里其实有 3 个层次，你要分开：

层次 1：能执行测试程序

- 例如 nestest 一类 CPU 测试
- 不一定需要完整 PPU 渲染
- 更强调 CPU、Bus、Cartridge 映射正确

层次 2：能启动简单游戏 ROM

- 至少能跑到主循环
- 能正确处理 Reset / NMI / VBlank
- 能显示基本画面

层次 3：能正常玩游戏

- 渲染正确
- 输入正确
- 时序稳定
- Mapper 支持足够
- 可能还要音频

你现在最现实的目标是：

- 先拿下层次 1
- 然后冲击一个最简单的 NROM 小游戏/演示 ROM

---------

2. 你现在已经有了哪些关键基础

按课程进度看，你已经有了这些非常重要的东西：

- CPU 基本指令框架
- 寻址模式
- 栈、分支、ADC/SBC、移位、BIT、BRK/RTI
- IRQ/NMI 基础
- DMA 基础
- PPU 最小寄存器骨架
- \$2000/\$2002/\$2006/\$2007 的最小模型
- OAM 与 OAM DMA
- Cartridge / mirroring 基础认识
- 整机 clock 的基础调度思路

这说明你离“第一款 ROM”不是很远了。

差的不是方向，而是闭环程度。

---------

3. 真正还差的第一大块：CPU 指令覆盖率

这是最直接的一块。

虽然你已经实现了很多指令，但真 ROM 一上来往往会用到更完整的 6502 指令集。
所以你要问自己：

- 常用官方指令是不是已经覆盖齐？
- 标志位行为是否稳定？
- 周期和页面翻转时机是否大体正确？
- JSR/RTS/BRK/RTI/IRQ/NMI 是否能稳定协作？

当前阶段最现实的要求不是：

- 一口气补全所有非法指令

而是：

- 先把官方常用指令补到足够跑简单 ROM

---------

4. 第二大块：Cartridge 映射必须真的闭环

这部分如果不闭环，ROM 根本谈不上“启动”。

你至少要确保：

- iNES 头解析正确
- PRG ROM 映射正确
- CHR ROM / CHR RAM 读写正确
- mirroring 正确传递给 PPU
- NROM 这种最基础 Mapper 能稳定工作

换句话说：

- 先别想着一开始就支持一堆 Mapper
- 先把 NROM 做扎实

因为你的第一款 ROM 大概率就应该选最简单的 NROM 目标。

---------

5. 第三大块：PPU 不需要完整，但必须够 ROM 启动

这是很多初学者容易误判的地方。

你不是说：

- “必须先完整实现所有 PPU 渲染，再跑 ROM”

而是要做到：

- PPU 至少足够支撑 ROM 的启动逻辑

通常这意味着至少要有：

- \$2000/\$2002
- VBlank 进入/离开
- NMI 触发链
- \$2006/$2007 的最小读写协议
- Name Table / Palette / CHR 基础访问
- OAM / DMA 基础

因为很多 ROM 启动时会做的事就是：

- 等待 VBlank
- 写调色板
- 写 Name Table
- 写 OAM
- 配置滚动/控制寄存器

所以“能跑第一款 ROM”的关键不是完整像素渲染，而是：

- PPU 协议层足够可用

---------

6. 第四大块：整机时序必须至少自洽

你现在已经有 NES::clock() 的雏形了。
这很好，因为跑 ROM 不是只靠模块功能，还靠它们能不能一起工作。

你至少要保证：

- PPU 一直推进
- CPU 按节拍推进
- NMI 来源于 PPU 状态
- DMA 能正确阻塞 CPU
- reset -> init -> vblank -> nmi -> 主循环 这条链是通的

对第一款 ROM 来说，不要求你现在就 cycle-perfect。
但要求你：

- 整机时间关系不能乱

---------

7. 第五大块：输入通常不是第一阻塞项，但很快就会需要

很多 ROM 即使没有输入也能跑到标题画面。

所以输入不是最先阻塞你的东西。

但只要你想从“启动 ROM”走到“能互动”，就得补：

- \$4016/\$4017
- 手柄 strobe
- 串行读取按键状态

所以输入通常不是“第一优先级”，但它是“第一批后续优先级”。

---------

8. 第六大块：渲染输出层

这个也要分清：

- “PPU 内部状态能正确更新”
- 和
- “你把它画到窗口上”

是两回事。

第一款 ROM 的早期阶段，你甚至可以先不追求漂亮窗口，而是先做到：

- PPU 内部缓冲可验证
- Name Table / Palette / OAM 状态合理
- 甚至先导出 debug 结果或简单 framebuffer

也就是说：

- 先让模拟器核心正确
- 再考虑前端显示方式

---------

9. 所以当前最合理的路线是什么

我建议你现在按这个顺序推进：

1. 锁定目标 ROM 类型
- 先选最简单的 NROM ROM
- 不要一开始碰复杂 Mapper 游戏
2. CPU 官方常用指令补齐
- 先为“能启动简单 ROM”服务
- 不必现在强迫自己全非法指令
3. 把 Cartridge + NROM + mirroring 闭环做稳
- PRG/CHR/Reset Vector/PPU 访问都要通
4. 把 PPU 协议层做够启动
- 重点是 VBlank/NMI/寄存器/VRAM 访问，不是完整渲染器
5. 验证 ROM 是否进入初始化主循环
- 能不能等到 VBlank
- 能不能收到 NMI
- 能不能正常写 PPU 数据
6. 再补输入和更完整渲染
- 从“能启动”走向“能玩”

这是最现实的路线。

---------

10. 当前阶段最容易走错的路

你现在最该避免 3 个误区：

误区 1：一头扎进完整 PPU 渲染

- 会淹死在细节里
- 而且未必是当前启动 ROM 的第一阻塞项

误区 2：继续无限补零散 CPU 指令但不做整机闭环

- 你会一直觉得“写了很多，但还是跑不起来”

误区 3：过早追求高精度 Mapper / APU

- 对第一款 ROM 来说通常太远了

---------

11. 这节课的真正结论

你现在离“第一款真实 ROM”差的不是某一个神秘大模块，而是这几个闭环是否完成：

- CPU 常用官方指令闭环
- NROM / Cartridge 映射闭环
- PPU 协议层闭环
- VBlank / NMI / DMA 时序闭环
- 最小整机 clock 闭环

只要这几个闭环成型，你就能从“学硬件”进入“跑程序”。
