第28课：Name Table 镜像与 Cartridge Mirroring 

这节课是很多 NES 初学者第一次真正意识到： 

  - “地址空间里看起来有 4 张 Name Table” 
  - 但机器里并不一定真的有 4 张独立背景表 
     

 先给核心结论： 
 
  - PPU 地址空间中看起来有 4 个 Name Table 区块 
  - 但标准 NES 主机本体通常只提供 2KB VRAM 
  - 而 4 个 Name Table 总共需要 4KB 
  - 所以其中一些区域必须镜像复用 
  - 具体怎么镜像，取决于卡带硬件的 mirroring 方式 
     

 这就是为什么“背景地图”不是简单 4 张独立表。 
 
---------

  1. 先看地址范围 
      

 PPU 背景 Name Table 相关地址通常是： 
 
  - 0x2000-0x23FF：Name Table 0 
  - 0x2400-0x27FF：Name Table 1 
  - 0x2800-0x2BFF：Name Table 2 
  - 0x2C00-0x2FFF：Name Table 3 
     

 每张 1KB，总共 4KB。 
 
 但问题来了： 
 
  - NES 主机常规只给 2KB 的 Name Table VRAM 
  - 所以不可能真的独立容纳 4 张 1KB 表 
     

 于是必须让其中两两共享底层存储。 
 
---------

  2. 什么叫“镜像” 
      

 镜像不是“把数据复制一份”。 
 
 镜像真正的意思是： 
 
  - 两个不同地址 
  - 最终映射到同一块底层内存 
     

 举个最简单的例子： 
 
  - 地址 A 写进去一个值 
  - 地址 B 读出来也能看到同一个值 
  - 因为它们底下其实是同一块存储 
     

 这就叫镜像。 
 
 所以镜像本质上是： 
 
  - 多个地址别名指向同一份物理数据 
     
---------

  3. Name Table 镜像为什么由卡带决定 
      

 这点非常重要。 
 
 你可能本能会以为： 
 
  - “既然是 PPU 的 VRAM，不就该由 PPU 自己决定吗？” 
     

 但 NES 不是这样。 

 Name Table 的镜像方式通常由： 

  - 卡带板电路 
  - 也就是 Cartridge / Mapper 所属硬件 

来决定。 

 所以镜像不是纯 PPU 内部逻辑，而是： 

  - PPU 地址空间访问 
  - 经过卡带硬件规则 
  - 最终落到哪一块 Name Table VRAM 
     

 这就是为什么 mirroring 属于 cartridge 元数据的重要组成部分。 
 
---------

  4. 最常见的两种：Vertical 和 Horizontal 
      

 这是你当前必须先搞懂的两种。 
 
  ### Vertical Mirroring 

 可以先记成： 
 
  - NT0 和 NT2 共享 
  - NT1 和 NT3 共享 
     

 也就是： 
 
 0x2000 -> A 
 0x2400 -> B 
 0x2800 -> A 
 0x2C00 -> B 
 
 所以是： 
 
  - 左右列不同 
  - 上下复用 
     
  ### Horizontal Mirroring 

 可以先记成： 

  - NT0 和 NT1 共享
  - NT2 和 NT3 共享

 也就是：

 0x2000 -> A
 0x2400 -> A
 0x2800 -> B
 0x2C00 -> B

 所以是：

  - 上下行不同
  - 左右复用

 你不一定现在就要把“水平/垂直”与滚屏方向完全绑定死记，但至少要把这两个映射图背下来。

---------

  5. 为什么会有这个设计

 因为 NES 很抠硬件资源。

 它不直接提供 4KB 独立 Name Table VRAM，而只给 2KB，剩下靠地址映射技巧解决。

 而卡带硬件可以选择不同镜像方式，让不同游戏更适合自己的滚屏需求。

 所以 mirroring 是一个非常典型的：
 
  - 用映射规则弥补硬件资源不足 
  - 并给不同游戏提供不同布局策略 
     

 的设计。 
 
---------

  6. 为什么它会影响滚屏 
      

 背景滚动本质上是在“跨 Name Table 看地图”。 
 
 如果 4 张 Name Table 真的都独立，那背景地图就会比较直接。 
 但因为有镜像： 
 
  - 某些区域其实是同一块底层存储 
  - 所以滚动跨过去时，看到的是“另一张独立地图”还是“同一张地图的镜像”，取决于 mirroring 方式 
     

 这就是为什么 mirroring 会直接影响背景滚动体验。 
 
 你现在先不需要深究具体滚屏效果，只要知道： 
 
  - mirroring 决定背景地图空间如何拼起来 
     
---------

  7. 你当前代码里已经开始碰到这件事了 
      

 你在 D:/code/rust/nes_core/src/bus.rs 里已经有了： 
 
  - Mirroring::Horizontal 
  - Mirroring::Vertical 
  - Mirroring::FourScreen 
  - nametable_index(addr) 
     

 这说明你已经在实现“PPU 地址空间到实际 VRAM 的映射”了。 

 这是非常对的方向。 

 你现在需要理解的是： 

  - 0x2000-0x2FFF 只是 PPU 看见的地址空间 
  - vram: [u8; 0x1000] 或其中某部分，才是你程序里的真实底层存储 
  - nametable_index(addr) 就是在做“从逻辑地址到物理存储索引”的翻译 
     

 也就是说，Bus/PPUMemory 正在扮演“镜像规则执行者”。 
 
---------

  8. FourScreen 是什么 
      

 这是第三种你先有印象即可： 

  - 有些卡带自己额外提供更多 VRAM 
  - 这样 4 张 Name Table 可以真的独立存在 
  - 不再需要普通的 Horizontal/Vertical 两两镜像 
     

 这就是 FourScreen。 
 
 当前阶段你知道它的意义就够了： 
 
  - FourScreen = 卡带额外给资源，让 4 张背景表都独立 
     
---------

  9. 这节课最容易混淆的点 
      
  ### 误区 1：镜像等于复制 

 不是。 
 镜像是多个地址映射到同一份底层存储，不是复制 2 份数据。 
 
  ### 误区 2：Name Table 一定有 4 张独立表 

 不是。 
 地址空间看起来像有 4 张，但实际常常只有 2 张的物理空间。 

  ### 误区 3：mirroring 是 PPU 自己决定的 

 通常不是。 
 它往往是卡带硬件/Mapper 决定的。 
 
---------

---------

  11. 这节课的最简总结 
       
  - Name Table 地址空间看起来有 4KB 
  - 标准 NES 往往只有 2KB 底层 VRAM 
  - 所以必须镜像 
  - 镜像是地址映射，不是数据复制 
  - mirroring 方式通常由 Cartridge/Mapper 决定 
  - 最常见的是 Horizontal 和 Vertical
