第37课：一款 NES ROM 真正启动时，CPU、PPU、Cartridge 分别在做什么

这节课很重要，因为你已经从“单模块理解”走到了“整机协作”。
现在我们把它们合成一次真正的启动图景。

先给核心结论：

- ROM 启动不是 CPU 自己一个人开始跑
- 而是 Cartridge 提供程序数据与映射规则
- CPU 从 Reset Vector 找到入口并开始执行
- PPU 同时按自己的时序初始化和运行
- 游戏程序再逐步初始化 RAM、PPU、输入、精灵、滚动等状态

所以“启动一款游戏”本质上是：

- 多硬件在同一时间轴上的协同启动

---------

1. Cartridge 在启动前已经决定了很多事

你前面学过 .nes / iNES / Header / PRG / CHR / Mapper。
现在你要把它们放回启动现场看。

Cartridge 在一开始就提供了至少这些东西：

- PRG ROM：CPU 要执行的程序
- CHR ROM / CHR RAM：PPU 要使用的图形数据
- Mirroring：背景 Name Table 的镜像方式
- Mapper 规则：CPU / PPU 地址空间如何映射到卡带资源
- 可能还有 SRAM、IRQ 能力等

所以在 CPU 真正执行第一条指令之前，Cartridge 已经决定了：

- CPU 读 0x8000+ 实际会读到什么
- PPU 读 Pattern Table 时会读到什么
- Reset Vector 是否能被正确读到
- 背景镜像方式是什么

这就是为什么我们一直说：

- Cartridge 不是“游戏资源包”
- 它是“硬件板 + 数据”

---------

2. CPU 启动时最关键的一步：读 Reset Vector

CPU reset 后不会从 PRG 的第一个字节开始跑。
它会去读：

- 0xFFFC
- 0xFFFD

这两个地址里的值组成一个 16 位地址，这就是 Reset Vector。

于是启动流程变成：

1. CPU reset
2. Bus 按地址映射把 0xFFFC/FFFD 转到 Cartridge 的 PRG 区域
3. Cartridge 提供这两个字节
4. CPU 得到真正入口地址
5. 从那个地址开始执行第一条指令

所以启动的第一条指令是否正确，极度依赖：

- PRG 映射是否正确
- Mapper 是否正确
- Reset Vector 是否能被正确读到

---------

3. PPU 并不是“等 CPU 初始化完再开机”

这是一个非常重要的观念。

你不能把启动过程想成：

- CPU 先完全初始化
- 然后 PPU 才开始工作

更真实的理解是：

- CPU 和 PPU 都在运行
- 只是游戏程序通常会在早期阶段主动去“配置 PPU”

也就是说：

- PPU 本来就在那
- 只是它一开始的状态还没有被游戏设置成最终想要的样子

所以启动时，游戏程序通常要做很多和 PPU 有关的初始化工作。

---------

4. 游戏程序启动后通常会做什么

你现在可以把“ROM 真正启动后的早期程序”理解成一组初始化动作：

1. 设置 CPU 状态
2. 清理或初始化 RAM
3. 设置栈
4. 配置 PPU 寄存器
5. 可能等待若干次 VBlank，让 PPU 稳定
6. 初始化调色板、Name Table、OAM
7. 初始化游戏对象状态
8. 进入正式主循环

这就是为什么游戏不是一开机就“立刻正常画面全出”。

中间有一段很重要的“引导初始化期”。

---------

5. 为什么很多程序启动时会先关 PPU 或暂时不显示

因为启动早期很多数据还没准备好。

比如：

- 背景数据还没写好
- 调色板还没设好
- 精灵表还没准备好
- 滚动参数还没初始化

这时候如果直接正常显示，画面会很乱。

所以游戏程序通常会：

- 先配置寄存器
- 在适当时机慢慢把数据写进去
- 再打开显示相关位

这就是为什么 \$2000/$2001 在初始化阶段非常重要。

---------

6. 为什么要等待 VBlank 再初始化图形数据

因为图形相关数据往往要通过 PPU 寄存器窗口写入：

- \$2006/$2007
- \$2003/$2004
- DMA 等

而你已经知道：

- 这些操作最好发生在 VBlank 安全窗口

所以很多游戏启动时会出现这样的节奏：

- reset 后先跑初始化代码
- 等待一次或几次 VBlank
- 在这些窗口中逐步把画面资源写进 PPU
- 最后再进入正式游戏主循环

这就是“为什么很多 ROM 一启动先像在忙准备工作”。

---------

7. CPU、PPU、Cartridge 在这个阶段分别扮演什么角色

可以这样总结：

Cartridge

- 提供程序与图形数据
- 决定映射规则
- 决定 mirroring
- 提供 Reset Vector

CPU

- 执行 PRG ROM 里的初始化代码
- 配置系统各部件
- 驱动游戏逻辑
- 与 PPU/输入/卡带交互

PPU

- 按时序持续运行
- 接收 CPU 对寄存器和显存的配置
- 最终按设置输出画面
- 在 VBlank 时通过状态/NMI 和 CPU 协作

这三者缺一不可。

---------

8. 你现在终于能理解“跑游戏”不只是 CPU 指令正确

就算 CPU 指令实现都对了，如果以下任一项不对，ROM 仍然可能跑不起来：

- Reset Vector 映射错误
- PRG 映射错误
- PPU 状态时序错误
- 
- VBlank / NMI 触发错误
- OAM DMA 行为错误
- mirroring 错误

所以“能跑游戏”的本质是：

- 多模块协同正确
- 不是单个 CPU 核心足够强

---------

9. 启动阶段最常见的“看起来像死机”的原因

从模拟器角度看，很多“游戏不启动”其实不是程序真坏了，而是某个关键启动协作点错了，例如：

- CPU 没有从正确 reset vector 起跑
- PPU VBlank/NMI 没正常来
- 初始化循环在等 $2002 某位变化，但那位永远不变
- Mapper 没映射对，导致入口代码本身就错

所以以后你调试 ROM 启动失败时，第一反应不要只是盯 CPU 指令，而要去看：

- Cartridge 映射
- PPU 状态
- NMI / VBlank
- 初始化循环在等什么

---------

10. 这节课的最简总结

- Cartridge 决定程序数据、图形数据和映射规则
- CPU reset 后从 Reset Vector 找到真正入口
- PPU 同时也在按自己的时序运行
- 游戏程序启动后会先初始化 RAM、PPU、OAM、调色板等
- 常常会等待 VBlank 再提交关键图形数据
- 所以“启动一款游戏”本质上是 CPU、PPU、Cartridge 的协同启动
