第35课：最小渲染循环，CPU、PPU、DMA 如何在 master clock 上一起推进

这节课的目标，是把你前面所有零散知识真正串成“整机循环”。

先给核心结论：

- NES 不是“CPU 跑完一帧，再让 PPU 画一帧”
- 而是 CPU、PPU、DMA 都在同一条 master clock 时间线上一起推进
- 只是它们推进速度不同、职责不同、还会互相影响

所以最小整机循环的本质是：

- 统一时钟驱动多个硬件组件同步前进

---------

1. 为什么必须有 master clock 这个概念

你现在已经知道：

- CPU 有自己的周期
- PPU 有自己的周期
- DMA 会抢总线
- NMI/IRQ 会在特定时刻发生

如果没有一条统一时间轴，你就没法回答这些问题：

- PPU 什么时候进入 VBlank？
- CPU 是在 VBlank 前还是后改了寄存器？
- DMA 抢总线时，CPU 此刻是不是正准备取指？
- NMI 是在这一拍还是下一拍被 CPU 采样到？

所以必须有一个更底层的统一节拍：

- master clock

它不一定要直接暴露给所有模块，但你的整机思维里必须有它。

---------

2. 最小的推进关系是什么

在你当前课程阶段，最重要的近似关系就是：

- 每 1 次 CPU 时钟，大约对应 3 次 PPU 时钟

所以整机推进的最小思路通常像这样：

master clock += 1
PPU 先走 1 拍
每逢第 3 拍，CPU 走 1 拍
DMA 若活跃，则在 CPU 时钟点抢占 CPU

这就是你现在 D:/code/rust/nes_core/src/lib.rs 已经开始做的事情。

---------

3. 为什么 PPU 必须几乎每拍都走

因为 PPU 是持续扫描器。

它不会说：

- “先等 CPU 做完一些逻辑”
- “再慢慢画图”

相反，它一直在：

- 前进 dot
- 推进 scanline
- 在某时刻进入 VBlank
- 在某时刻清 VBlank
- 在某时刻可能触发 sprite zero hit

所以 PPU 是整机里最“持续工作”的组件之一。

---------

4. CPU 为什么不能每拍都执行一条完整指令

因为 CPU 的“1 次 clock”不等于“执行完 1 条指令”。

你现在的模型里已经开始体现这点了：

- CPU 有 cycles
- 如果 cycles > 0，这拍只是消耗剩余周期
- 只有当剩余周期用完时，CPU 才会进入下一次取指/执行流程

这很重要，因为它说明：

- CPU 不是按“指令个数”推进
- 而是按“时间”推进

这正是你前面从 step 走向 clock 的根本变化。

---------

5. DMA 为什么要插进这条循环里

因为 DMA 不是一段独立脚本，它也在时间线上发生。

它的特点是：

- 只要 DMA 活跃
- CPU 在自己的 clock 点上就不能正常取指
- 这一拍会被 DMA 消耗掉

所以整机循环里，CPU 这边的逻辑会变成：

1. 当前 CPU 这拍轮到我了吗？
2. 如果轮到了，先看 DMA 是否抢占
3. 如果 DMA 抢占，就不执行正常 CPU 指令
4. 如果没有，再看中断
5. 最后才执行 opcode

这就是为什么 DMA 不应该做成普通函数拷贝，而必须插进整机主循环。

---------

6. 最小整机循环应该按什么顺序想

当前阶段你可以先用这个思维顺序：

1. master_clock += 1
2. PPU clock()
3. 更新 CPU 能看到的 NMI 线状态
4. 如果到了 CPU 节拍：
- 先处理 DMA
- 再处理中断
- 最后 CPU 正常推进一拍

这就是你现阶段最合理的“系统主循环骨架”。

---------

7. 为什么要在 PPU 之后更新 NMI 线

因为 NMI 来源于 PPU 状态。

也就是说：

- 不是 CPU 自己凭空知道 NMI 来了
- 而是 PPU 先推进到某个时刻
- 比如进入 VBlank
- nmi_line 发生变化
- 然后 CPU 在自己的时间点观察到它

所以 NMI 的顺序必须是：

- PPU 先变状态
- CPU 后看见状态

这就是硬件时序，不是普通函数调用。

---------

8. 你现在已经能理解为什么“先后顺序”这么敏感

同样是一个 clock() 循环，如果顺序写成：

- 先 CPU
- 再 PPU

和写成：

- 先 PPU
- 再 CPU

在很多边界时刻，结果就可能不同。

例如：

- VBlank 刚开始那一拍
- NMI 什么时候被 CPU 锁存
- 某个寄存器写入到底影响当前帧还是下一部分画面

这说明模拟器的难点不是只有逻辑，还有调度顺序。

---------

9. 当前阶段你最小可接受的整机架构

你现在最合适的结构就是：

NES / Host

- 持有 CPU
- 持有 Bus
- Bus 里再持有 PPU、DMA、Cartridge 等
- 提供一个总的 clock() 入口

然后：

- NES::clock() 推进 master clock
- 推进 PPU
- 同步 NMI 线
- 每 3 拍推进一次 CPU
- CPU 内部再决定这拍是不是被 DMA / interrupt 抢占

这条线你现在已经走在正确路上了。

---------

10. 为什么这节课是一个分水岭

前面你学的是：

- CPU 指令
- 地址映射
- 中断
- PPU 寄存器
- DMA

这些都像单点知识。

从这一课开始，你要真正形成的是：

- 系统时钟调度思维

也就是说，你以后看任何 NES 行为，都要先问：

- 它发生在什么时间点？
- 谁先推进？
- 谁后观察？
- 这个行为消耗几拍？
- 会不会影响别的设备？

这才是整机模拟器的核心视角。

---------

11. 这节课的最简总结

- master clock 是整机统一时间轴
- PPU 几乎每拍都推进
- CPU 只在较慢节拍上推进
- CPU 的一拍也不等于一条完整指令
- DMA 会在 CPU 节拍上抢占总线
- NMI/IRQ 都是时间线上被采样的事件
- 整机主循环的本质是“统一时钟调度多个硬件”

课后问题

1. 为什么不能把整机理解成“CPU 先跑完一段，再让 PPU 跑一段”？
2. 为什么 DMA 必须插进 CPU 的 clock 流程，而不是做成独立的一次性函数？
3. 为什么 NMI 的正确顺序必须是“PPU 先变状态，CPU 后观察到”？
4. 这节课之后，你觉得 NES::clock() 最核心的职责是什么？

你先回答这 4 个问题。答完后，下一课我讲：

第36课：真正的游戏主循环在干什么，为什么很多 NES 游戏看起来像是在“等 NMI”。

---

1. CPU和PPU是在同一个时间轴上共同推进的，而不是各自轮流跑。
2. DMA也是NES时序的一部分，在数据帮运过程中，时序依旧是在一步步推进，而不是一次性就完成。
3. 因为NMI的触发依赖于PPU，只有PPU更新了NMI状态，CPU才能观察到。
4. NES::clock的核心职责就是调度各个模块的时序，让它们在一个统一的时间轴上共同推进。
