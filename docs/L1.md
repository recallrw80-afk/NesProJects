## 第 1 课

 今天只做一件事：把 NES 看成一个“会运行的系统”，而不是一堆陌生缩写。你先不用会汇编，也不用懂电路。先把整体图建立起来，后面学 CPU、PPU 才不会迷路。

 这节课主要对应文档里的架构部分 doc/doc.md:17 和模块划分 doc/doc.md:35。 

1. 模拟器到底在模拟什么 

 最容易犯的错，是把模拟器理解成“把 ROM 文件翻译成电脑代码执行”。

 不是。 

 模拟器做的事情更接近： 

  1. 在内存里维护一套“虚拟硬件状态”。 
  2. 按照 NES 真机规则推进这些状态。 
  3. 在合适的时候把画面、声音、输入接出来。 

 也就是说，模拟器的核心不是“翻译代码”，而是“扮演硬件”。
 你可以先记住这个公式： 
 模拟器 = 硬件状态 + 时钟推进 + 输入输出 
 这里的“硬件状态”包括： 

  - CPU 的寄存器现在是什么 
  - PPU 的显存和寄存器现在是什么 
  - 卡带当前映射到哪一页 ROM
  - 手柄当前按了哪些键 
  - 当前是不是到了一帧结束 
    
2. NES 里最重要的 6 个角色

 文档已经把核心模块列出来了 doc/doc.md:37。 
 你先把它们当成一个小型计算机系统： 

  - CPU
    负责执行游戏程序。它会不断“读指令 -> 执行 -> 修改状态”。 
  - PPU
    图像芯片。它不关心“游戏逻辑”，它只关心“这一像素该画什么”。 
  - APU
    音频芯片。负责生成方波、三角波、噪声等声音。 
  - Cartridge 
    卡带。里面装着游戏程序和图形数据，也可能带有额外硬件。 
  - Mapper 
    卡带上的“扩展控制器”。它决定 CPU/PPU 看到的是哪一段 ROM。 
  - Bus
    总线。CPU 并不是直接摸到所有硬件，而是通过总线访问地址，再由总线把请求转发给正确模块。 

 你可以先用一句话记： 
 CPU 负责想，PPU 负责画，APU 负责叫，Cartridge 负责装数据，Mapper 负责切页，Bus 负责转发。

3. 什么叫“总线” 
   

 这是第一节最关键的概念。 

 你平时写应用时，可能觉得“访问内存”就是：

 let x = mem[0x1234];

 但 NES 里不是这样。
 0x1234 只是一个“地址”，它背后到底指向什么，要看地址映射规则。

 这个项目的 bus.c 已经把这个思想写得很清楚了：core/bus.c:31 

 它先判断地址落在哪个区间：

  - 0x0000 到 0x1FFF：RAM 
  - 0x2000 到 0x3FFF：PPU 寄存器 
  - 0x4014：DMA 
  - 0x4016、0x4017：手柄 
  - 更高地址很多会转到卡带 

 也就是说： 

 CPU 读一个地址 -> Bus 判断这个地址属于谁 -> 再把读写请求转给那个模块

 看这个实现最直观：core/bus.c:55 

 它不是直接读一个大数组，而是： 

  - RAM 区就读 __ram 
  - PPU 区就调用 ppu_bus_read 
  - 手柄区就读控制器移位寄存器 
  - 卡带区就调用 cart_read 

 这就是“内存映射 I/O”。 

 你现在先把这个概念吃透： 

 地址不是存储本身，地址只是入口。 

4. 为什么 Cartridge 和 Mapper 要分开 

 这两个词很容易混。 

 先粗暴理解：

  - Cartridge 是整张卡带 
  - Mapper 是卡带里面的“银行切换芯片” 
    

 为什么要有 Mapper？

 因为 NES CPU 能直接看到的程序地址空间有限，但游戏可能很大。 
 所以卡带会想办法说： 

 “你现在访问 0x8000 时，先给你看第 0 页 PRG；过一会儿我再切到第 3 页。” 

 这就像一本超厚书，桌上一次只能摊开两页，Mapper 负责翻页。 

 所以后面你会看到： 

  - 没有 Mapper，很多游戏根本装不下
  - Mapper 不只是“存储”，而是“动态映射规则” 
    
5. 前端、核心、接口，为什么要分层
   

 文档用的是 Frontend / Core / Bridge 思路 doc/doc.md:17。 

 这个设计非常对，尤其适合你用 Rust 重写。

 分层后： 

  - Core 只管 NES 逻辑 
  - Frontend 只管窗口、键盘、音频设备 
  - Bridge 只管双方怎么交换数据 
    

 好处很大： 

  - 核心能做单元测试 
  - 将来能换 SDL、GLFW、网页前端 
  - 核心逻辑不会和平台 API 缠在一起
    

 这也是为什么文档借鉴了 RetroArch/Libretro 的结构 doc/doc.md:5。 

6. 这一份 C 项目里，一帧是怎么跑出来的 
   

 现在看核心循环，你就会对“模拟器不是普通业务程序”有直觉了。 

 看这里：core/libretro.c:27

 它每次 retro_run() 做的事大概是：

  1. 读取输入 
  2. 把输入状态写进总线 
  3. 一直循环推进硬件，直到 PPU 产出一帧图像 
  4. 把这一帧交给前端显示 
  5. 把这一帧的音频样本交给前端播放
     

 中间最重要的是这段逻辑：core/libretro.c:41 

 核心意思是：

  - 如果 DMA 正在工作，就先推进 DMA
  - 否则推进 CPU 一步
  - 每推进一次 CPU，再推进 3 次 PPU
  - 如果 PPU 触发了 NMI，通知 CPU 
  - 如果卡带 Mapper 触发了 IRQ，通知 CPU 
    

 你现在不用先搞懂 NMI 和 IRQ 的细节，只要知道： 

 NES 不是 CPU 单线程瞎跑，而是多个硬件按时序一起推进。 

 这就是“时序”的雏形。 

7. 用 Rust 该怎么想这个系统 
   

 如果你用 Rust 写，不要一上来就写 opcode 表。 
 先把“系统模型”立起来。 

 最小心智模型应该是： 

```rust
pub struct Nes { 
    cpu: Cpu, 
    ppu: Ppu, 
    apu: Apu, 
    bus: Bus, 
    cart: Cartridge, 
} 
```

 然后你会逐渐发现一个现实问题： 

 Bus 需要访问 Cpu/Ppu/Cart，而这些模块有时又需要回头访问 Bus。

 这就是模拟器里常见的“互相引用”问题。 

 所以后面在 Rust 里，我们会认真设计： 

  - 谁拥有谁 
  - 谁只保留状态 
  - 谁通过函数参数访问别人 
  - 怎么避免乱用 Rc<RefCell<_>> 
    

 第 1 课你先不用解决它，但要知道：

 Rust 写模拟器，真正难的往往不是语法，而是“怎么把硬件关系建模得清楚”。 

8. 这节课你必须记住的 4 句话 
   
     模拟器不是翻译器，而是硬件状态机。 
     
     CPU 不直接访问模块，而是通过总线访问地址。 
     
     地址不是数组下标，而是“映射入口”。 
     
     NES 的运行不是 CPU 单独跑，而是 CPU、PPU、APU、DMA、Mapper 按时序协同推进。 

一个最小类比 

 把 NES 想成一家小工厂： 
  - CPU 是调度员，负责看指令单 
  - PPU 是画图车间，负责出图像 
  - APU 是音频车间，负责出声音 
  - Cartridge 是仓库，里面放程序和素材 
  - Mapper 是仓库管理员，决定今天拿哪一批货 
  - Bus 是厂区道路，所有请求都要经过它分流
    

 这个类比不精确，但对入门很有效。 

10. 课后小练习 

 你先不要写代码，先自己回答这 3 个问题： 
  1. 为什么说 0x8000 不是“某个固定数组下标”？ 
  2. 为什么模拟器里需要 Bus，而不是让 CPU 直接调用 PPU、Cartridge？ 
  3. 为什么 Mapper 不是可有可无的东西？ 

 你能把这 3 个问题说顺，第 1 课就算真的听明白了。
