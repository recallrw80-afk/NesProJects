第 7 课：CPU 执行循环 

 这节课是 CPU 模拟的核心。 
 前面几课你学的是零件，现在开始看“CPU 一条指令到底怎么跑”。 
 先记一句总纲： 
 CPU 执行一条指令，本质上就是：取指 -> 找操作数 -> 执行 -> 更新状态 

  1. 最小执行模型 

```rust
fn clock(&mut self) {
    let opcode = self.read(self.pc);
    self.pc += 1;
    let instruction = decode(opcode);
    let operand = resolve_addressing(instruction.addr_mode); 
    execute(instruction, operand); 
} 
```

这不是完整代码，但很接近核心思想。 
 这里有 3 个关键动作： 

  1. 从 PC 指向的位置读 opcode 
  2. 根据 opcode 知道“这是什么指令 + 用什么寻址模式” 
  3. 执行指令，修改寄存器、标志位、内存、PC 
     
    4. 什么是取指

 取指 = 根据 PC 去内存里拿下一条指令的第一个字节 

 比如： 

  - PC = 0xC000 
  - 读到 0xA9 

 如果你后面查表知道： 
  - 0xA9 = LDA immediate 

 那 CPU 就知道： 
  - 这条是 LDA 
  - 寻址模式是 IMM 

 然后 CPU 再继续读后面的操作数。 

 所以： 

 opcode 只是“指令编号” 

 CPU 拿到 opcode 后，才知道后面该怎么解释。 

    3. 为什么 PC 会不断变化

 因为 CPU 是顺着内存往后读的。 
 最简单情况： 

  - 取完 opcode，PC += 1 
  - 如果这条指令还有 1 字节操作数，再 PC += 1 
  - 如果还有 2 字节操作数，再继续前进 

 比如： 

 LDA #\$10 

 可能占 2 个字节： 

  - 第 1 字节：opcode 
  - 第 2 字节：立即数 0x10 

 执行完后，PC 就会指向下一条指令。 
 但如果是跳转类指令，比如： 

  - JMP 
  - JSR 
  - RTS 
  - BEQ 

 那 PC 就不一定只是“顺着加”，而可能被直接改到别处。 
 所以你可以把 PC 理解成： 
 默认顺序前进，但某些指令会强行改道 

    4. 解码到底在做什么 

 解码 = 根据 opcode 查出这条指令的规则 

 通常至少要知道三件事： 

  - 指令是什么，比如 LDA、STA、ADC 
  - 寻址模式是什么，比如 IMM、ABS、IZY 
  - 周期数大概是多少 

 很多模拟器会做一张 opcode 表。 
 因为 6502 的真正执行逻辑不是“看名字猜”，而是： 

 opcode -> instruction metadata 

 比如某个表项可能表示： 

 opcode 0xA9 => LDA, IMM, 2 cycles 

 你现在先不用背 opcode，只要知道 CPU 每次都是先查表，再执行。 

    5. 执行阶段到底干什么 

 执行阶段通常包含这几类事：
  - 读操作数 
  - 做运算 
  - 写回结果 
  - 更新标志位 
  - 更新 PC 
  - 可能压栈/弹栈 
  - 可能触发总线读写 

 举个最简单的例子： 
 LDA #\$00 
 执行时： 

  1. 通过 IMM 找到立即数 
  2. 把值写进 A 
  3. 更新 Z 
    4. 更新 N 

 所以指令执行的本质不是神秘黑箱，而是： 
 一条规则明确的状态变换 

    6. 为什么“寻址模式”和“指令逻辑”最好分开 

 因为很多指令共享寻址模式。 
 比如 LDA/ADC/AND/CMP 都可能用 IMM/ABS/ABX/... 
 如果不拆开，你会写出很多重复代码： 

  - 在 LDA 里算一遍地址 
  - 在 ADC 里再算一遍 
  - 在 AND 里再算一遍 

 这样很快就乱。 
 所以更清晰的结构是： 

  1. 先根据寻址模式找到“值在哪”或“地址在哪” 
    2. 再执行指令自己的动作 

 也就是： 

 寻址负责找对象，指令负责做动作 

 这就是你上一题第 6 题答对的地方。 

    7. 一个完整的小例子：LDA $20 

 假设： 
  - PC = 0x8000 
  - 0x8000 处是 LDA zp 的 opcode 
  - 0x8001 处是操作数 0x20 
  - 内存地址 0x0020 的值是 0x3C 

 执行流程： 
  1. CPU 在 0x8000 取到 opcode 
  2. PC 前进 
  3. 解码得知：这是 LDA，寻址模式 ZP0 
  4. 再读 0x8001，得到操作数 0x20 
  5. 把 0x20 当零页地址 0x0020 
  6. 去总线读 0x0020 
  7. 读到值 0x3C 
  8. A = 0x3C 
    9. 根据结果更新 Z 和 N 

 注意，这里真正进入 A 的不是 0x20，而是地址 0x0020 里的内容 0x3C。 

    8. 一个完整的小例子：STA $2000

 再看存储类指令。 
 STA $2000
 流程： 

  1. 取 opcode 
  2. 解码：STA + ABS 
  3. 再读两个字节，拼成地址 0x2000 
    4. 把 A 的值写到地址 0x2000 

 这时就要注意了： 
  - 0x2000 在 NES 里通常不是普通 RAM 
  - 很可能是 PPU 寄存器区域 

 所以 STA $2000 的效果可能不是“改一个数组元素”，而是“通知 PPU 某个控制寄存器被写了”。 

 这正是模拟器里总线和内存映射的重要性。 

    9. CPU 执行不是“一条指令一次函数调用”那么简单 

 从高层看可以这么理解。 
 但从模拟精度上说，还涉及“周期”。 
 因为真实 CPU 不是瞬间执行完一条指令，而是花若干个 clock cycle。 
 所以很多模拟器会把 CPU 设计成： 

  - 当前指令还有多少 cycle 没跑完 
  - 如果为 0，就取下一条指令 
  - 否则继续消耗 cycle 

 你现在先不用被这个细节绊住。 
 先建立第一层理解： 
 逻辑上是一条条执行，时序上是一拍拍推进 
 后面讲更精确的 CPU/PPU 同步时，这个 distinction 会很重要。 

    10. Rust 里你将来很可能会这样建模 

 你以后大概率会有： 

```rust
struct Instruction {
    name: Op,
    mode: AddrMode,
    cycles: u8,
} 
```

再配一张表： 

```rust
const OPCODES: [Instruction; 256] = ... 
```

然后：

```rust
fn cpu_step(&mut self) {
    let opcode = self.bus_read(self.pc);
    self.pc = self.pc.wrapping_add(1);
    let inst = OPCODES[opcode as usize];
    let operand = self.resolve(inst.mode);
    self.execute(inst, operand);
} 
```

这只是骨架，但方向就是这个方向。 
    11. 这一课最重要的认知 

 你现在要记住： 
 CPU 不是“直接执行文本指令”，而是不断读取字节流，并按 opcode 规则解释这些字节 
 也就是说，ROM 在 CPU 眼里首先不是“汇编代码文本”，而是一串字节。 
 只是这些字节刚好能按 6502 规则解释成指令。 
 这是从“看汇编”切到“实现 CPU”的一个关键转变。 

  12. 这节课你必须记住的 5 句话
      
      CPU 执行循环的核心是：取指、解码、执行。 
      
      PC 指向下一条要取的指令字节。 
      
      opcode 告诉 CPU“这条指令是什么、怎么找操作数”。 
      
      寻址模式负责找到数据，指令逻辑负责处理数据。 
      
      CPU 从底层看是在解释字节流，不是在读文本汇编。
