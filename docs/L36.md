第36课：真正的游戏主循环在干什么，为什么很多 NES 游戏看起来像是在“等 NMI”

这节课是你从“模拟器作者视角”转到“游戏程序作者视角”的关键一步。

先给核心结论：

- 许多 NES 游戏的主循环，不是现代程序那种“无限 while 然后每次自己主动刷新画面”
- 它更像：
- 平时做逻辑准备
- 到了 VBlank/NMI 时再进行本帧关键图形更新
- 所以很多游戏看起来像是在“等 NMI”

这不是偷懒，而是硬件决定的最佳工作方式。

---------

1. 为什么游戏不能随便什么时候都改 PPU

你前面已经学过：

- PPU 不是静止 framebuffer
- 它在一帧里持续扫描输出
- 非 VBlank 时乱改 PPU 相关数据，可能导致画面异常

所以程序员不能像现代 API 那样想：

- “我想改画面，就立刻改”

在 NES 上，更合理的做法是：

- 先准备好要更新的数据
- 等到 VBlank 到来
- 在那段安全窗口里把关键更新写进 PPU

这就是“等 NMI”的根本原因。

---------

2. NMI 在游戏里扮演什么角色

你可以把 NMI 理解成：

- 每帧一次的时机通知

只要游戏启用了 $2000.bit7 的 VBlank NMI：

- PPU 一进 VBlank
- CPU 就收到 NMI
- 游戏代码就知道：
- “现在可以做这一帧的显示更新了”

所以 NMI 在很多游戏里不只是中断，它更像：

- 每帧同步信号
- 一帧结束 / 新帧准备开始的时钟点

---------

3. 一个非常典型的 NES 游戏结构

很多游戏逻辑可以粗略理解成两部分：

主线程平时做的事

- 读输入
- 更新游戏逻辑
- 算角色位置
- 算滚动值
- 准备要写入 PPU 的数据

NMI 里做的事

- 真正把这些结果写进 PPU 寄存器 / VRAM / OAM
- 完成本帧画面更新

所以主线程和 NMI 的关系常常是：

- 主线程负责“算”
- NMI 负责“提交到画面”

---------

4. 为什么不是所有逻辑都写在 NMI 里

因为 VBlank 时间很短。

也就是说：

- 不是你想在 NMI 里干多少就干多少
- 它只是一个有限窗口

如果你把太多复杂逻辑塞进 NMI：

- 来不及
- 容易打乱节奏
- 影响帧稳定性

所以合理分工是：

- 日常逻辑在主循环里慢慢做
- 到 NMI 时只做关键显示更新

这和现代“逻辑线程 / 渲染提交”有点像，但原因更硬件化。

---------

5. 为什么会出现“主循环像是在等 NMI”

因为很多游戏会这样组织：

1. 主循环先做一部分逻辑
2. 然后等待一个“这一帧更新完成”的信号
3. NMI 到来时完成画面提交
4. 主循环再继续下一帧逻辑

从表面看就像：

- 程序一直在等 NMI
- NMI 一来才继续动

本质上它是在用 NMI 做：

- 帧同步

所以“等 NMI”其实可以理解成：

- 等待下一帧安全更新时机

---------

6. 这和你前面学的 Sprite Zero Hit 有什么关系

两者都和“时机”有关，但用途不同：

NMI / VBlank

- 一般用于整帧更新时机
- 是“帧级别”的同步点

Sprite Zero Hit

- 常用于一帧中间的某个位置
- 是“扫描到某个分界线”的同步点

所以你现在要开始把时机分层理解：

- 帧级同步：VBlank / NMI
- 帧内局部同步：Sprite Zero Hit

这会对你以后理解 HUD 分割、卷轴切换特别重要。

---------

7. 为什么这会反过来影响模拟器设计

因为如果模拟器的：

- VBlank 时机
- NMI 触发时机
- $2002 标志变化
- PPU/CPU 时序关系

做错了，游戏程序里的“等 NMI”逻辑就会出问题。

也就是说：

- 游戏程序并不是随便写的
- 它是在依赖硬件节奏工作
- 模拟器如果节奏不对，游戏主循环就会乱

所以你现在学的这些，不只是理论，而是决定游戏能不能正常跑的核心。

---------

8. 你现在可以把游戏主循环想成什么

最简单的脑图是：

while running:
处理输入
更新游戏对象状态
生成本帧需要提交给 PPU/OAM 的数据
等待 NMI
下一帧继续

而 NMI 里做：

在 VBlank 中把准备好的图形状态提交给 PPU

这就是 NES 游戏“主循环像在等 NMI”的本质。

---------

9. 为什么“等待 NMI”不是阻塞式偷懒，而是节奏控制

从现代程序员角度，你可能会本能地觉得：

- “等中断”听起来像阻塞、浪费

但在 NES 上，这恰恰是正确的硬件节奏感。

因为问题不是 CPU 忙不忙，而是：

- 你什么时候能安全地和 PPU 交互
- 你什么时候应该切到下一帧逻辑

所以等待 NMI 更像：

- 等待下一帧节拍
- 而不是“傻等事件”

---------

10. 这节课的最简总结

- 游戏不能随便在任何时刻更新 PPU
- VBlank / NMI 给了游戏一个每帧一次的安全更新时机
- 主循环通常负责逻辑计算
- NMI 通常负责把结果提交到 PPU
- 所以很多 NES 游戏看起来像是在“等 NMI”
- 本质上它是在做帧同步
