第 10 课：先做最小 CPU，第一批指令怎么实现 

 前面我们一直在铺地基。 
 从这节开始，终于进入“真正写 CPU”这件事。 
 但注意，我们不会一上来就实现 151 条官方指令和各种边角细节。 
 那样你会直接被淹死。 
 正确做法是： 
 先做一个最小可运行 CPU，再逐步补全。
 这节课的核心目标是理解： 

  1. 第一版 CPU 最少需要哪些组成部分 
  2. 一条指令在代码里怎么落地 
  3. 为什么“实现所有指令”不是第一步 

---------

1. 最小 CPU 需要哪些东西 

 如果只为了让 CPU 开始跑起来，第一版最少要有： 
  - 寄存器 
      - A/X/Y 
      - SP 
      - PC 
      - P 
  - 基本总线读写 
      - read(addr) 
      - write(addr, val) 
  - reset 
      - 从 0xFFFC/0xFFFD 读入口地址 
  - 指令执行框架 
      - 取指 
      - 解码 
      - 执行 
  - 一小批高频指令 
      - 加载/存储 
      - 转移 
      - 递增递减 
      - 跳转/分支 
      - 栈操作的最基础部分 

 所以第一版 CPU 并不是“完整 CPU”，而是： 
 能证明执行框架跑通的最小子集 

---------

2. 为什么不能一开始就全实现 

 因为 CPU 实现的困难不在于“写很多 case”，而在于： 
  - 状态位容易错 
  - 寻址模式容易错 
  - PC 更新容易错 
  - 栈行为容易错 
  - 周期行为容易错 
  - 和总线交互容易错 

 如果你一上来全写，很容易出现： 
  - 代码写了很多
  - 但完全不知道错在哪 

 所以更稳的路线是： 
  1. 先实现骨架 
  2. 先实现最好理解的一小批指令 
  3. 用简单 ROM 或测试程序验证 
  4. 再逐步扩展 

 这和写编译器、解释器、虚拟机是一样的思路。 

---------

3. 第一批最适合实现的指令 

 我建议你第一批优先实现这些： 
  - LDA 
  - LDX 
  - LDY 
  - STA 
  - TAX 
  - TAY 
  - TXA 
  - TYA 
  - INX 
  - INY 
  - DEX 
  - DEY 
  - JMP 
  - JSR 
  - RTS 
  - BEQ 
  - BNE 
  - NOP 

 为什么是这批？ 
 因为它们覆盖了几种最核心能力： 

  - 读数据 
  - 写数据 
  - 寄存器搬运 
  - 顺序执行 
  - 改变流程 
  - 最基本循环控制 

 有了这些，你已经能跑起不少简单逻辑。 

---------

4. 一条指令在代码里到底长什么样 

 先看一个最简单的： 
 LDA #$10 
 它的语义可以写成： 

```rust
self.a = value; 
self.set_zn(self.a); 
```

 你会发现，指令实现其实很朴素。 
 真正难的常常不是这两行，而是： 

  - value 是怎么来的？ 
  - PC 怎么前进？ 
  - Z/N 怎么更新？ 
    

 所以很多指令实现都能压缩成一句“状态变换”。 

 比如： 

 INX 

```rust
 self.x = self.x.wrapping_add(1); 
 self.set_zn(self.x); 
```

 TAX 

```rust
 self.x = self.a; 
 self.set_zn(self.x); 
```

 STA 

```rust
 bus.write(addr, self.a); 
```

 这时你就会发现前面分离“寻址模式”和“指令逻辑”的价值了。 

---------

5. 先做一个好用的小工具：更新 Z/N 
     

 很多指令执行完都要更新： 

  - Z 
  - N 
    

 所以你通常会抽一个辅助函数： 

```rust
fn set_zn(&mut self, value: u8) { 
    self.set_flag_z(value == 0); 
    self.set_flag_n((value & 0x80) != 0); 
} 
```

 这样： 

  - LDA 
  - LDX 
  - LDY 
  - TAX 
  - TXA 
  - INX 
  - DEX 
    

 等很多指令都能复用。 

 这类小辅助函数在 CPU 实现里非常重要，能大幅减少重复和出错率。 

---------

6. LDA 怎么实现 
     

 假设寻址模式层已经帮你取到了值。 

 那么 LDA 的核心动作就是： 

```rust
fn op_lda(&mut self, value: u8) { 
    self.a = value; 
    self.set_zn(self.a); 
} 
```

 这说明： 

  - LDA 只负责“把值装进 A，并更新标志” 
  - 它不需要知道值最初来自立即数、零页还是绝对地址 
    

 这就是职责拆分。 

---------

7. STA 和 LDA 有什么不同 
     

 LDA 是读。 

 STA 是写。 

 所以如果 LDA 是： 

 self.a = value; 

 那 STA 通常是： 

```rust
 bus.write(addr, self.a); 
```

 这里要注意一个很关键的差异： 

  - 对 LDA，你更关心“值是什么” 
  - 对 STA，你更关心“地址是什么” 
    

 也就是说，有些指令更依赖“取值”，有些更依赖“算地址”。 

 这会影响你后面设计 resolve() 返回什么。

---------

8. JMP 怎么实现 
     

 JMP 是流程控制里最简单的： 

```rust
 self.pc = target_addr;
```

 就这么简单。 

 但前提是： 

  - 你得先根据寻址模式拿到正确目标地址 
  - 如果是 JMP abs，目标就是操作数字面给的地址 
  - 如果是 JMP ind，目标还得先间接取出来 
    

 所以 JMP 指令本身不复杂，复杂的是目标地址怎么求。 

---------

9. BEQ / BNE 为什么很值得早做 
     

 因为它们能让程序形成条件分支和循环。 

 比如： 

  - BEQ：如果 Z = 1，跳转 
  - BNE：如果 Z = 0，跳转 
    

 这意味着只靠： 

  - 比较/加载结果设置标志 
  - 再用分支看标志 
    

 程序就能控制流程。 

 你现在先不用实现完整比较指令，也能先理解这套机制： 

 先靠某条指令留下状态位，再靠分支指令读取状态位 

 这是 CPU 流程控制的基础套路。 

---------

10. JSR / RTS 为什么重要 
      

 这是子程序调用机制。 

 JSR 

  - 调用子程序 
  - 本质上要先把“返回地址”压栈 
  - 再把 PC 改到目标地址 
    

 RTS 

  - 从栈里把返回地址取回来 
  - 再继续执行 
    

 这意味着如果你要支持稍微像样一点的程序，栈机制就绕不开。 

 不过因为 JSR/RTS 比 JMP 多了“压栈/弹栈”，所以你做实现时要格外小心： 

  - 压栈顺序 
  - 返回地址是哪一个 
  - PC 调整规则 
    

 这些地方都很容易出 off-by-one 错误。 

---------

11. 第一版 CPU 不求什么 
      

 第一版不要追求： 

  - 精确周期 
  - 所有非法指令 
  - 极端边界行为 
  - 所有中断细节 
  - 一次通过所有测试 ROM 
    

 第一版只求： 

 执行框架正确，核心寄存器变化正确，基础读写正确，最小流程能跑 

 先“能走”，再“走得准”。 

---------

12. 一个推荐的实现顺序 
      

 如果是你现在开始写 Rust，我建议按这个顺序补： 

  1. reset() 
  2. fetch opcode 
  3. IMM/ZP0/ABS/REL 这几种最基础寻址 
  4. LDA/LDX/LDY 
  5. STA 
  6. TAX/TAY/TXA/TYA 
  7. INX/DEX/INY/DEY 
  8. JMP 
  9. BEQ/BNE 
  10. 栈辅助函数 
  11. JSR/RTS 
      

 为什么 JSR/RTS 放后面？ 

 因为它们对栈和 PC 的处理最容易把你搞乱。 
 先把前面的线性执行模型跑通，再碰子程序跳转更稳。 

---------

13. Rust 里第一版指令表可以长什么样 
      

 你未来很可能会有类似这样的结构： 

```rust
enum Op { 
    Lda, 
    Sta, 
    Tax, 
    Inx, 
    Jmp, 
    Beq, 
    Bne, 
    Nop, 
} 
```

 再配： 

```rust
struct Instruction { 
    op: Op, 
    mode: AddrMode, 
    cycles: u8, 
} 
```

 这样 CPU 每次： 

  1. 取 opcode 
  2. 查 Instruction 
  3. 根据 mode 解析操作数 
  4. 根据 op 执行动作 
     

 这就是最典型、最清楚的写法之一。 

---------

14. 这节课最重要的认知 
      

 你现在要形成这个感觉： 

 一条指令的实现通常不难，难的是把“取值、算地址、改标志、改 PC、读写总线”这些环节接对。 

 所以写 CPU 时，别把注意力都放在“这条指令看起来像不像汇编书上的定义”。 

 更重要的是问： 

  - 它输入什么状态？ 
  - 输出什么状态？ 
  - 它依赖值还是地址？ 
  - 它会不会改 PC？ 
  - 它会不会改 Z/N/C/V？ 
  - 它会不会读写栈？ 
    

 这才是模拟器视角。 

---------

15. 这节课你必须记住的 6 句话 
      
  1. 第一版 CPU 只需要最小可运行子集，不需要一口吃完整套 6502。 
  2. LDA/STA/JMP/BEQ 这类指令已经能覆盖很多核心行为。 
  3. 很多指令实现很短，真正复杂的是它们和寻址/状态位/总线的配合。 
  4. set_zn() 这种辅助函数非常重要。 
  5. JSR/RTS 的难点不在跳转，而在栈和返回地址处理。 
  6. 先跑通骨架，再补精度，是实现 CPU 的正确顺序。
