第 9 课：CPU 是怎么真正开始跑起来的
 前面你已经知道： 

  - ROM 里有 PRG 
  - CPU 通过地址空间访问 PRG 
  - CPU 有 PC 
  - CPU 靠“取指 -> 解码 -> 执行”工作 

 现在还差最后一块拼图： 

 CPU 上电后，第一条指令到底从哪来？ 

 答案不是“从 ROM 开头开始”。 
 真正答案是： 

 CPU 会先去读一个特殊地址里的启动向量，再把读出的值装进 PC。 

 这就是 reset vector。 

---------

1. CPU 不会“从文件第一字节开始执行” 

 这是很多初学者最自然、也最错误的直觉。 
 因为你平时看源代码、脚本、可执行文件，很容易觉得： 

  - 程序总得有个“开头” 
  - 那大概就是文件最前面开始执行 

 但 NES 不是这么工作的。 
 对 CPU 来说，它根本不认识“文件头”“文件开头”这种概念。 
 它只认识地址空间。 
 所以 CPU 启动时不会说： 

  - “去 .nes 文件第几个字节开始跑” 

 而是说： 
  - “去某个固定地址读启动向量” 
  - “把读到的地址写进 PC” 
  - “从那个地址开始取指” 

 这是一种彻底的硬件思维。 

---------

2. 什么是 reset vector 

 reset vector = CPU 复位后要去的起始地址指针 
 对 6502/NES 来说，reset 时 CPU 会读取： 

  - 0xFFFC 
  - 0xFFFD 

 这两个地址里的内容 

 然后把它们拼成一个 16 位地址，装入 PC。 

 比如： 
  - mem[0xFFFC] = 0x00 
  - mem[0xFFFD] = 0xC0 

 那 reset 后： 
  - PC = 0xC000 

 然后 CPU 才从 0xC000 开始取第一条指令。 
 所以你现在先记住： 
 0xFFFC/0xFFFD 里放的不是程序代码，而是“程序入口地址”。 

---------

3. 为什么要用 vector，而不是写死起点 

 因为这样 CPU 设计更灵活。 
 如果不靠 vector，而是硬编码成“永远从某个固定地址开始”，那程序布局就很死。 
 现在靠 vector，ROM 作者就能决定： 

  - 实际入口在哪 
  - reset 后该跳到哪段初始化代码 

 所以你可以把 vector 理解成： 
 启动时查的一张跳转指针 
 除了 reset vector，后面你还会见到： 

  - NMI vector 
  - IRQ/BRK vector 

 也就是说，不只是 reset，连中断入口也是通过固定地址里的“指针”来找的。 

---------

4. reset vector 为什么会落在高地址 

 因为 CPU 地址空间顶端这几处地址被专门保留给向量区。 
 对于 6502，这是一种硬件约定。 

 你现在不用从电路历史去理解它，先把它当成架构规则： 

  - 0xFFFA/0xFFFB：NMI vector 
  - 0xFFFC/0xFFFD：Reset vector 
  - 0xFFFE/0xFFFF：IRQ/BRK vector 

 这也解释了为什么： 

 PRG ROM 的高地址区非常重要 

 因为这些向量地址最终必须能从 CPU 地址空间里读到。 
 所以卡带的 PRG 映射必须保证这些高地址是有效的。 

---------

5. reset 时真正发生了什么 

 先讲简化版，不抠所有细节。 
 reset 后，CPU 至少会做这些事： 

  1. 初始化一部分内部状态 
  2. 设置/重置某些标志位 
  3. 从 0xFFFC/0xFFFD 读出 reset vector 
  4. 把它写入 PC 
  5. 从 PC 指向的地址开始取第一条指令 

 你现在可以把它想成： 

```rust
fn reset(&mut self) { 
    self.a = 0; 
    self.x = 0; 
    self.y = 0; 
    // 其他状态初始化... 

    let lo = self.bus.read(0xFFFC) as u16; 
    let hi = self.bus.read(0xFFFD) as u16; 
    self.pc = (hi << 8) | lo; 

} 
```

 这不一定是最终最精确实现，但概念上就是这样。 

---------

6. 为什么 reset vector 会和 ROM 映射强相关 

 因为 0xFFFC 和 0xFFFD 不在主机内部 RAM 区，也不在 PPU 区， 
 它们通常落在： 

  - 0x8000..0xFFFF 

 也就是卡带 PRG 区。 

 所以当 CPU reset 时，实际上是在通过 Bus 去卡带那边取数据。 

 这说明三件事已经连起来了： 

  - CPU 知道要读 0xFFFC/0xFFFD 
  - Bus 知道这个高地址该转发给 Cartridge 
  - Cartridge/Mapper 知道这个地址该映射到 PRG ROM 的哪一段 

 这就是为什么我前面一直说： 
 模拟器不是写几个模块，而是让模块之间的硬件关系跑通 

---------

7. 为什么 NROM-128 会涉及镜像 

 这是个很好理解 reset vector 的例子。 

 有些简单卡带，比如 NROM-128，只有 16KB PRG ROM。 
 但 CPU 高地址程序区是： 

  - 0x8000..0xFFFF 

 这有 32KB 

 那怎么办？ 

 就把这 16KB PRG 映射两次：

  - 0x8000..0xBFFF 
  - 0xC000..0xFFFF 
    

 都指向同一份 PRG 数据 

 这样高地址区依然有效，reset vector 所在的 0xFFFC/0xFFFD 也就能读到正确内容。 

 这就是为什么“镜像”不仅出现在 RAM，也会出现在卡带映射上。 

---------

8. ROM 文件里的 PRG 和 CPU 地址空间不是一回事 

 这个概念你现在必须再强化一次。 
 ROM 文件里的 PRG 是“文件中的一段字节数据”。 
 CPU 地址空间里的 0x8000..0xFFFF 是“CPU 能访问到的高地址区域”。 
 这两者之间不能直接画等号，必须经过： 

  - Cartridge 解析 
  - Mapper 映射 

 也就是说： 
 PRG 是原料 
 映射后的 CPU 地址区是运行时视图 
 reset vector 就恰好体现了这一点。 
 CPU 读的是地址 0xFFFC， 
 但这个地址最终对应 PRG 里的哪个偏移，取决于 Mapper 规则。 

---------

9. 第一次取指的完整链路 

 你现在应该能顺下来这条链了： 

  1. 加载 .nes 文件 
  2. 解析 header 
  3. 拿到 PRG 数据 
  4. 建立 Cartridge 和 Mapper 
  5. CPU reset 
  6. CPU 通过 Bus 读取 0xFFFC/0xFFFD 
  7. Bus 把请求转发到 Cartridge 
  8. Cartridge 根据映射规则返回两个字节 
  9. CPU 拼出 PC 
  10. CPU 从这个 PC 开始取第一条 opcode 

 如果这 10 步你脑子里能连起来，那前面几课已经真正串起来了。 

---------

10. NMI 和 IRQ vector 先顺便认识一下 

 虽然我们今天不讲中断细节，但现在顺手认识非常值得。 

 CPU 遇到不同类型事件时，不是跳到写死代码，而是也会去读固定地址里的向量： 

  - NMI vector：告诉 CPU，NMI 发生时该去哪里 
  - IRQ/BRK vector：告诉 CPU，IRQ 或 BRK 时该去哪里 

 所以“vector”这个思想很统一： 
  - reset 查一次 
  - 中断也查一次 

 你可以理解成： 
 这些高地址就是 CPU 的“紧急入口地址表” 

---------

  11. Rust 实现里 reset 最少要做什么 
      

 你后面写 Rust 时，第一版大概会有： 

```rust
impl Cpu { 
    pub fn reset(&mut self, bus: &mut Bus) { 
        self.sp = 0xFD; 
        self.p = 0x24; // 或类似约定值 
        let lo = bus.read(0xFFFC) as u16; 
        let hi = bus.read(0xFFFD) as u16; 
        self.pc = (hi << 8) | lo; 
    } 
} 
```

 这里你先不用纠结每个初始标志位是不是百分百精确。 
 现在最重要的是理解： 

 reset 的关键动作是读取 reset vector 并设置 PC 

 如果这一步没做对，CPU 后面根本不可能正常起跑。 

---------

  12. 这节课最重要的认知 
      

 你现在要真正摆脱一种应用程序员直觉： 

 不是“程序从文件开头开始”。 

 而是： 

 程序从 CPU 通过 reset vector 找到的入口地址开始。 

 这句话非常重要。 
 后面你看 ROM、看 CPU 调试日志、看启动流程时，都会反复用到。 

---------

  13. 这节课你必须记住的 6 句话 

- CPU 启动时不是从文件第一字节开始执行。 

- reset 后，CPU 会读取 0xFFFC/0xFFFD。 

- 这两个字节拼出的 16 位值会写入 PC。 

- PC 指向的地方，才是第一条指令的位置。 

- 向量地址通常位于 CPU 高地址区，因此依赖 PRG 映射正确。 

- reset vector、NMI vector、IRQ vector 本质上都是“入口地址指针”。
