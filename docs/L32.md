第32课：从 OAM 到真正显示，PPU 怎么把“精灵描述”变成“屏幕上的 sprite 像素”

这节课的目标是把你脑子里的两件事接起来： 

1. OAM 里只有精灵描述信息 
2. 屏幕上最后却真的出现了像素 


那中间到底发生了什么？ 

先给核心结论： 

- OAM 不直接给像素 
- PPU 会在渲染过程中，根据当前扫描位置，从 OAM 挑出“这一行会出现哪些 sprite” 
- 再根据 tile 编号和属性，去 Pattern Table 取图形位 
- 最后把这些结果组合成真正的 sprite 像素 

所以 sprite 显示是一个“查表 + 筛选 + 组合”的过程。 

---------

1. 先回忆一下屏幕不是整张一起画的 


你前面已经学过： 

- PPU 不是瞬间生成整帧 
- 它是按时间推进的 
- 画面是一行一行扫出来的 


这就意味着： 

- PPU 在当前这一时刻，并不关心“全部 64 个 sprite 的全部像素” 
- 它只关心： 
- 当前扫描线 
- 当前像素位置 
- 有哪些 sprite 会在这里出现 


这就是 sprite 渲染的出发点。 

---------

2. 第一步：从 OAM 里筛选“当前扫描线相关的 sprite” 


OAM 有 64 个 sprite 条目。 
但当前扫描线不可能同时关心全部 64 个。 

所以第一步是： 

- 看每个 sprite 的 Y 
- 判断它是否覆盖当前扫描线 
- 如果覆盖，就把它选出来 


你现在先把它理解成： 

- “扫描线过滤” 


也就是说，PPU 会先问： 

- 哪些 sprite 出现在这一行？ 

---------

3. 为什么只看 Y 还不够 


因为知道 sprite 在这条扫描线上还不够，还得知道： 

- 这一行是 sprite 图块里的第几行像素 


例如一个 sprite 从屏幕 Y=40 开始，当前正在画 scanline=43，那就说明： 

- 当前要取的是这个 sprite 的第 3 行图形数据


所以第二步其实是： 

- 根据当前扫描线和 sprite 的 Y 
- 算出这个 sprite 该取 tile 的哪一行 


这会直接受到：

- 垂直翻转 
- sprite 高度模式 


的影响。 

当前阶段你先重点理解“要先算 tile 内部行号”。 

---------

4. 第二步：根据 tile 编号去 Pattern Table 取图 


筛选出 sprite 后，PPU 就知道： 

- 用哪一个 tile 编号 
- 取 tile 的第几行 


然后去 Pattern Table 里取对应图形位。 

这里你要记住一个核心事实： 

- Pattern Table 里存的是像素位平面数据，不是最终颜色 


所以取出来的还不是“最终彩色像素”，而只是这一行 sprite 的图形位信息。 

---------

5. 第三步：结合属性字节修正显示 


这时候属性字节开始真正发挥作用： 

- 如果水平翻转，像素取样顺序要反过来 
- 如果垂直翻转，tile 行号计算要反过来 
- 
- 如果 palette 选择不同，之后查色时要用不同 sprite palette 
- 如果优先级不同，和背景混合时规则不同 


这说明属性字节不只是一个“附属元数据”，而是直接参与像素生成过程。 

---------

6. 第四步：生成 sprite 的像素值 


到这里，PPU 才能得到一个更接近最终结果的东西： 

- 这个 sprite 在当前扫描线上的若干像素 
- 每个像素对应某个 sprite 调色板索引 
- 还带着透明/不透明、优先级等信息 


注意这里依然不是“最终上屏颜色”，因为还没和背景混合。 

---------

7. 第五步：和背景竞争同一个屏幕像素 


最终某个屏幕位置上可能既有： 

- 背景像素 
- 精灵像素 


这时候就要根据规则决定： 

- 显示背景 
- 还是显示 sprite 
- 还是 sprite 因透明而让背景透出来 


所以 sprite 渲染不是“单独生成完就结束”，它还要参与和背景的最终合成。 

这也正是为什么属性里的优先级很重要。 

---------

8. 为什么透明也很关键 


你可以把 sprite 像素理解成不总是“有颜色”。 

很多 sprite 像素其实是透明的。 
也就是说： 

- 虽然这个 sprite 覆盖了当前位置 
- 但这个具体像素可能不画任何东西 
- 背景就会透出来 


所以最终上屏不是“只要 sprite 在这里就一定盖住背景”，而是： 

- 还要看这个 sprite 当前像素是不是透明 

---------

9. 你现在该如何把整个过程压成一个最小模型 


当前阶段你不需要记一堆细节，只要记这条流水线： 

1. 从 OAM 里找出当前扫描线涉及的 sprite 
2. 算出每个 sprite 当前要取 tile 的哪一行 
3. 去 Pattern Table 取图形位 
4. 用属性字节处理翻转、palette、优先级 
5. 得到 sprite 像素 
6. 再和背景像素合成 


这就是“从 OAM 到 sprite 像素”的最小逻辑链。 

---------

10. 为什么这说明 OAM 是“描述表”而不是“图像表” 


因为 OAM 真正提供的是： 

- 放在哪 
- 用哪个 tile 
- 用什么显示规则 


而不是： 

- 每一个像素具体是什么颜色 


真正像素是 PPU 在渲染过程中临时算出来的。 

所以你现在应该更确信： 

- OAM 是描述 
- Pattern Table 是图形原料 
- 属性字节是渲染规则 
- 最终像素是运行时生成的 

---------

11. 当前阶段你不需要马上实现什么 


你现在还不需要立刻实现： 

- 每扫描线 8 sprite 限制 
- sprite zero hit 
- 精确优先级细节 
- 8x16 sprite 模式 
- 逐点像素混合 


这些都是后续专题。 

当前你先把“筛选 -> 取图 -> 属性修正 -> 像素生成 -> 与背景合成”的大流程理解清楚。 

---------

12. 这节课的最简总结 

- OAM 提供的是 sprite 描述，不是像素 
- PPU 会按当前扫描线，从 OAM 筛选相关 sprite 
- 再去 Pattern Table 取 tile 图形位 
- 再结合属性字节处理翻转、palette、优先级 
- 最后把 sprite 像素和背景像素合成到屏幕上
