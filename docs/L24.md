第24课：PPU 寄存器最小模型，\$2000-\$2007 到底在控制什么 

 这节课不讲完整渲染流程，只讲 CPU 能碰到的 PPU 门口寄存器。 
 你要先建立一个非常重要的概念： 

  - CPU 看不见 PPU 内部全貌 
  - CPU 只能通过少数几个寄存器去“操纵” PPU 
  - 所以这几个寄存器其实就是 CPU 和 PPU 之间的协议 

---------

1. 先把 \$2000-\$2007 当成 PPU 的控制面板 

 最小模型如下： 
  - $2000 PPUCTRL 只写
  - $2001 PPUMASK 只写
  - $2002 PPUSTATUS 只读
  - $2003 OAMADDR 只写
  - $2004 OAMDATA 可读可写
  - $2005 PPUSCROLL 只写
  - $2006 PPUADDR 只写
  - $2007 PPUDATA 可读可写

 你现在已经实际碰到过： 
  - \$2003/\$2004：OAM 读写 
  - 之后你一定会频繁碰到： 
      - $2000 
      - $2002 
      - \$2006/\$2007 

---------

2. \$2000 PPUCTRL 是“控制开关寄存器” 

 它主要负责告诉 PPU： 
  - 背景/精灵使用哪张 Pattern Table 
  - VRAM 地址自增步长 
  - NMI 是否开启 
  - 以及当前基础 Name Table 选择 

 现在你先不用背每一位，只要记住： 
  - 它是“控制 PPU 行为”的配置寄存器 
  - 后面讲到 NMI 和背景取数时，它会非常重要 

 特别是其中一个点你要提前记住： 
  - VBlank 时是否触发 NMI，和 \$2000 有关 

 这会把你前面学的中断和 PPU 真正连起来。 

---------

3. $2001 PPUMASK 是“显示开关寄存器” 

 它主要决定： 
  - 是否显示背景 
  - 是否显示精灵 
  - 左边几列是否裁剪 
  - 是否开启某些渲染输出效果 

 你现在可以先把它理解成： 
  - “画面显示层面的开关控制” 

 不是存数据，而是控制“显示哪些东西”。 

---------

4. \$2002 PPUSTATUS 是“状态寄存器” 

 这是非常关键的一个寄存器。 

 它会反映 PPU 当前状态中的某些信息，比如： 

  - 是否进入 VBlank 
  - 精灵相关标志位 

 而且它有一个特别重要的特点： 
  - 读它不仅是“获取状态” 
  - 读它还可能带副作用 

 例如以后你会学到： 
  - 读 \$2002 会清掉 VBlank 标志 
  - 还会影响 \$2005/\$2006 的写入锁存状态 

 这说明 PPU 寄存器不是普通变量，不是“读一下完全无副作用”。 

---------

5. \$2003/\$2004 你已经在用了 

 这两个你现在已经有实践感了： 
  - $2003：设置 OAM 地址 
  - $2004：对当前 OAM 地址读写数据 

 所以 CPU 要操作精灵内存时，不是直接： 

 ppu.oam[42] = x; 

 而是通过寄存器协议： 

  1. 写 $2003 
  2. 再读写 $2004 

 而 OAM DMA 其实也是另一条批量写 OAM 的专门通道。 

---------

6. $2005 PPUSCROLL 是滚动寄存器 

这个寄存器用来设置背景滚动位置。 

你先不要急着深入它的位级结构。 

当前只要理解： 

  - 背景不是固定死在屏幕上的 
  - 游戏可以卷动画面 
  - $2005 就是 CPU 告诉 PPU“滚动到哪里”的入口之一 

 后面你会发现它和 $2006 共享一套写入锁存逻辑，所以它不是一个简单的“写一个值就完了”的寄存器。 

---------

7. $2006 PPUADDR 是“PPU 内部地址寄存器的写入口” 

 这就是前面你已经理解过的核心点： 
  - PPU 有自己的地址空间 
  - CPU 不能直接访问 
  - CPU 要先通过 $2006 把目标地址告诉 PPU 

 而且 $2006 不是一次写完的，通常是： 
  1. 先写高字节
  2. 再写低字节 

 也就是说，CPU 是通过两次写寄存器，把一个 16 位 PPU 地址送进去。 
 这点非常重要。 

---------

8. $2007 PPUDATA 是“通过当前 PPU 地址进行数据读写的窗口” 

 一旦 \$2006 已经设置好了 PPU 内部地址，CPU 就可以通过 \$2007： 
  - 读 PPU 数据 
  - 写 PPU 数据

 也就是说： 
  - \$2006 负责“地址” 
  - \$2007 负责“数据”

 这俩合起来就像： 

 set address -> access data 

 所以你可以把它理解成 PPU 内部地址空间的一扇窗口。 

---------

9. 为什么 \$2005/\$2006 会让很多人头疼 

 因为它们不是普通寄存器，而是依赖“写入顺序”。 

 例如 $2006： 

  - 第一次写：高字节 
  - 第二次写：低字节 

 这就说明 PPU 内部需要记住： 
  - “这是第一次写，还是第二次写？” 

 也就是说，PPU 除了寄存器值本身，还需要维护一些写入状态机。 

这就是为什么以后实现 PPU 时，不能只放几个 u8 字段就完事。 

---------

10. 你现在最应该建立的最小寄存器分组  

 你可以先按功能把它们分 4 组： 
 控制组 

  - $2000 PPUCTRL 
  - $2001 PPUMASK 

 状态组 
  - $2002 PPUSTATUS 

 OAM 组 
  - $2003 OAMADDR 
  - $2004 OAMDATA 

 VRAM/滚动访问组 
  - $2005 PPUSCROLL 
  - $2006 PPUADDR 
  - $2007 PPUDATA 

 这样你脑子里会清晰很多。 

---------

11. 这节课和你当前代码怎么衔接 

 你现在的 D:/code/rust/nes_core/src/ppu.rs 只是实现了 OAM 那一小块。 
 这非常正常，因为在当前课程进度下，你最先需要的是： 

  - 让 OAM 存在 
  - 让 \$2003/\$2004 工作 
  - 让 DMA 有目标可写 

 下一步如果继续往前推进，最自然的方向就是： 
  1. 给 PPU 加最小寄存器字段 
  2. 先支持 \$2000/\$2002 
  3. 再建立 \$2006/\$2007 的地址-数据窗口模型 
  4. 最后再接 VRAM 存储 

 不要一上来就想完整渲染。 

---------

12. 这一课最关键的结论 

  - \$2000-\$2007 不是“几块普通内存” 
  - 它们是 CPU 和 PPU 之间的协议接口 
  - 有些寄存器写一次不够，要靠顺序 
  - 有些寄存器读会带副作用 
  - 这就是为什么 PPU 比普通 RAM 难实现得多
