第19课：CPU 中断、时序、周期
 
 这节是 CPU 阶段里最容易“代码先写出来了，但脑子里还没成图”的一课。 
 你现在正好卡在这个点上，所以这节不以指令为主，而是把 3 个概念一次讲透： 
 
  1. 中断是什么 
  2. 时序是什么 
  3. 周期是什么 
      

 先给你一句总定义： 
 
  - 中断：CPU 暂停当前主线，先去处理一件更紧急的事，处理完再回来 
  - 时序：这些事在“什么时候发生”，先后顺序是什么 
  - 周期：完成这些事要消耗多少个硬件节拍 
     
---------

  1. 先用你熟悉的编程世界类比 
      

 你可以把 CPU 想成一个单线程程序： 
 
 fetch -> decode -> execute -> fetch -> decode -> execute 
 
 正常情况下，它一直顺序执行。 
 
 中断就像： 
 
  - 不是你主动调用了某个函数 
  - 而是系统突然告诉你：“先停一下，现在必须先处理这个事件” 
     

 所以中断和普通函数调用最大的区别是： 
 
  - 函数调用：程序自己决定跳过去 
  - 中断：外部事件或特殊机制强制插入 
     
---------

  2. 中断到底打断了什么 
      

 它打断的是“当前正在执行的主线”。 
 
 比如 CPU 本来要继续执行： 
 
 LDA ... 
 STA ... 
 ADC ... 
 
 但这时如果来了 NMI，CPU 会： 
 
  1. 先把“我现在执行到哪了”保存起来 
  2. 把“我当前状态是什么”保存起来 
  3. 跳去执行中断处理入口 
  4. 执行完后再回来

 这个“保存起来”的动作，就是你现在已经实现过的： 
 
  - 压栈 PC 
  - 压栈 P 
  - 跳向量 
  - RTI 恢复 
     

 所以你已经把“机制”写出来了，现在只是要把“意义”想明白。 
 
---------

  3. 为什么必须保存 PC 和 P 
      

 因为 CPU 被打断时，如果不保存现场，它根本不知道怎么回来。 
 
 保存 PC 是为了知道： 
 
  - 处理完中断后，应该回到哪条指令继续 
     

 保存 P 是为了知道： 

  - 中断前的标志位状态是什么 
  - 比如 Z/N/C/I/V 这些状态不能丢 
     

 所以中断本质上就是： 
 
  - 暂停 
  - 存档 
  - 去处理别的事 
  - 读档回来 
     
---------

  4. NES 里你当前应该理解的三种“中断相关入口” 
      

 Reset 
 
  - 不是“打断当前执行” 
  - 而是“强制重新开始” 
  - 从 FFFC/FFFD 取入口地址 
     

 IRQ 
 
  - 普通可屏蔽中断 
  - 如果 I=1，CPU 不响应 
  - 向量是 FFFE/FFFF 
     

 NMI 
 
  - 不可屏蔽中断
  - 不看 I 
  - 向量是 FFFA/FFFB 
     

 你现在先把它们看成三种“跳转入口来源”就够了。 
 
---------

  5. BRK、IRQ、NMI 的关系 
      

 这三个很容易混。 
 
 你可以这样记： 
 
  - BRK：软件主动触发“像 IRQ 一样”的流程 
  - IRQ：硬件发起的普通中断 
  - NMI：硬件发起的高优先级中断 
     

 它们共享很多流程： 
 
  - 压栈 
  - 设置 I 
  - 跳向量 
     

 但也有关键区别：
 
 BRK 
 
  - 是指令 
  - 会表现成压入 PC+2 
  - 压栈状态时 B=1 
  - 用 FFFE/FFFF 
     

 IRQ 
 
  - 不是指令，是外部事件 
  - 压当前下一条待执行位置 
  - 压栈状态时 B=0 
  - 用 FFFE/FFFF 
     

 NMI 
 
  - 也是外部事件 
  - B=0 
  - 但走 FFFA/FFFB 
     

 所以你以后不要把 BRK 和 IRQ 当成同一个东西，只能说它们“很像”。 
 
---------

  6. 那么，中断到底是谁发出来的 
      

 在 NES 里，中断不是凭空来的，是别的硬件发给 CPU 的。 
 
 最典型的是： 
 
  - PPU 可以触发 NMI 
  - 某些 Mapper 可以触发 IRQ 
  - APU 也可能参与 IRQ 相关机制 
     

 所以 CPU 并不是宇宙中心。 
 它不是主动轮询一切，而是有时候会被别的硬件“叫停”。 
 
 这就是你前面一直在学的“总线系统思维”真正开始发挥作用的地方。 
 
---------

  7. 你现在“不理解中断”的根本原因通常不是代码，而是缺少时间观 
      

 高层语言里，你更习惯这样想： 
 
  - 我调用函数 
  - 我控制流程 
  - 一切按顺序执行 

 硬件里你必须补上另一半思维： 
 
  - 有些事情不是你调用的 
  - 它们是在某个时刻发生 
  - CPU 只是对那个时刻做出响应 
     

 所以中断本质上是“时间驱动的控制流变化”。 
 
 这句话你要记住。 
 
---------

  8. 什么叫时序 
      

 时序就是： 
 
  - 某件事先发生 
  - 某件事后发生 
  - 它们之间隔了多少个周期 
  - 某个信号在哪个时刻被采样 
     

 对于模拟器来说，时序不是玄学，它决定了模拟器是否像真机。 
 
 比如： 
 
  - CPU 什么时候检查 NMI pending 
  - 分支什么时候多 1 周期 
  - DMA 什么时候抢总线 
  - PPU 和 CPU 谁先走一步 
     

 这些全是时序问题。 
 
---------

  9. 什么叫周期 
      

 周期就是硬件运行的最小时间单位之一。 
 
 你现在写 CPU 时，最常见的是“指令周期”这个概念： 
 
  - LDA immediate 大约 2 cycles 
  - JSR 6 cycles 
  - RTI 6 cycles 
  - BRK 7 cycles 
     

 这里的意思不是“Rust 代码跑了几行”，而是： 
 
  - 真正的 6502 硬件完成这条指令，要消耗多少个 CPU 周期 
     

 所以周期是硬件时间，不是代码复杂度。 
 
---------

  10. 为什么你现在的 cpu_clock 这个名字其实不太准确 
       

 这里我要直接纠正你一下： 
 
 如果你这个函数“一次执行完整一条指令”，那它更像： 
 
  - step 
  - step_instruction 
  - execute_next_instruction 
     

 而不是真正的 clock
 
 因为真正的 clock 更接近： 
 
  - 一次只推进 1 个 CPU 周期 
  - 不一定完成一整条指令 
     

 这是后面做精确同步时非常关键的术语区别。 
 
 你现在如果函数行为还是“执行一整条指令”，那改名成 cpu_clock 反而更容易把自己带偏。 
 
 最稳妥的理解是： 
 
  - step: 执行一条指令 
  - clock: 推进一步硬件时钟 
     

 这两个以后最好分开。 
 
---------

  11. 为什么一条指令会有多个周期 
       

 因为 CPU 不是魔法，它得分步骤做事。 
 
 以一条需要读内存的指令为例，硬件上大致会经历： 
 
  1. 取 opcode 
  2. 读操作数 
  3. 计算地址 
  4. 访问内存 
  5. 写寄存器 / 改标志 
      

 不同指令复杂度不同，所以周期不同。 
 
 例如： 
 
  - INX 只改寄存器，很快 
  - JSR 要读地址、改 PC、压两次栈，周期更多 
  - BRK 要做额外压栈和跳向量，所以更慢 
     
---------

  12. 为什么有些情况下还会“额外多 1 周期” 
       

 你前面已经碰到过两个典型例子： 
 
  1. 分支成功 
      
  - 分支不成功，少做一步 
  - 分支成功，要真正改 PC，所以多 1 周期 
     
  2. 翻页 
      
  - 如果地址计算跨页，例如 0x12FF + 1 -> 0x1300 
  - 硬件处理会多一步 
  - 所以再加 1 周期 
     

 这就是“行为对了还不够，时间也得对”的最好例子。 
 
---------

  13. 为什么周期以后会变得极其重要 
       

 因为 NES 不是只有 CPU。 
 
 你已经知道它还有： 
 
  - PPU 
  - APU 
  - DMA 
  - Mapper 
     

 这些硬件不是“等 CPU 执行完一条指令再说”，而是都在时间线上一起跑。 
 
 最关键的一个粗略事实是： 
 
  - PPU 速度大约是 CPU 的 3 倍 
     

 所以你以后真正做整机模拟时，常见思路会变成： 
 
  - CPU 每走 1 个周期 
  - PPU 走 3 个周期 
  - 某些硬件在特定周期上改变状态 
  - 于是某一时刻可能触发 NMI、DMA 等事件 
     

 所以周期的真正价值是：让所有硬件共享同一条时间轴。 
 
---------

  14. 那中断和周期是怎么连起来的 
       

 这是本课最核心的一句： 
 
  - 中断不是“抽象地发生了” 
  - 中断是在某个具体时间点发生，并在某个具体时机被 CPU 观察到 
     

 也就是说： 
 
  - PPU 在某个时刻把 NMI pending 置上 
  - CPU 不是立刻瞬移过去 
  - 而是在自己的中断检查时机看到它 
  - 然后进入中断流程 
     

 所以： 
 
  - 中断是事件 
  - 周期是时间刻度 
  - 时序是 CPU 在哪一个刻度处理这个事件 
     
---------

  15. 你现在工程里应该怎么理解“pending” 
       

 你如果写了： 
 
  - nmi_pending: bool 
  - irq_pending: bool 
     

 那你其实已经在模拟一个重要概念了： 
 
  - 别的硬件不是直接改 CPU 的 PC 
  - 而是先发出“有中断请求了” 
  - CPU 在合适时机再响应 
     

 这就是 pending 的意义。 
 
 它不是“马上执行中断”，而是： 
 
  - 先记下来 
  - 等 CPU 检查时再处理
     
---------

  16. 为什么 NMI 和 IRQ 要分优先级 
       

 因为它们表示的重要程度不同。 
 
 一般你现在先记： 
 
  - 先检查 NMI 
  - 再检查 IRQ 
     

 原因很简单： 
 
  - NMI 不可屏蔽，优先级更高 
  - IRQ 还要看 I 标志 
     

 所以 CPU 的逻辑通常是： 
 
 if nmi_pending: 
 进入 NMI 
 else if irq_pending && !I: 
 进入 IRQ 
 else: 
 正常取指执行 
 
 这就是最基础的“中断仲裁”。 
 
---------

  17. 这一阶段你应该形成的架构认识 
       

 到这里，CPU 不再只是“执行 opcode 的机器”了。 
 它其实还承担两类职责： 
 
  1. 指令执行器 
      
  - 取指 
  - 解码 
  - 执行 
  - 改寄存器与标志 
     
  2. 时间线上会被外部事件影响的硬件 
      
  - 被 NMI/IRQ 打断 
  - 被 DMA 暂停 
  - 与 PPU/APU 协同推进
     

 所以从这一课开始，你要慢慢从“指令模拟器”转向“硬件组件模拟器”。 
 
 这一步非常重要。 
 
---------

  18. 你现在最容易混淆的 3 件事 
      
  1. step 和 clock 
      
  - step 更像“执行一条完整指令” 
  - clock 更像“推进一个硬件周期” 
     
  2. BRK 和 IRQ 
      
  - 流程像，但来源不同，B 位语义不同 
     
  3. “有中断请求” 和 “CPU 已经进入中断” 
      
  - pending 只是请求挂起 
  - 真正进入中断要等 CPU 检查时机 
     
---------

  19. 你现在不必立刻做到 cycle-accurate，但要分层 
       

 你当前阶段建议这样分层理解： 
 
 第一层：功能正确 
 
  - NMI/IRQ/BRK/RTI 跳转和恢复正确 
     

 第二层：指令周期正确 
 
  - 每条指令周期数基本正确 
  - 分支/翻页这些额外周期正确 
     

 第三层：整机时序正确 
 
  - CPU、PPU、APU、DMA 在同一时间轴协同运行 
     

 你现在大概刚进入第二层，正在往第三层门口走。 
 
---------

  20. 这节课你真正要带走的结论 
       

 不是“我会写 NMI 了”，而是： 
 
  - CPU 不是只会顺序执行指令 
  - 硬件世界里，流程会被时间事件打断 
  - 周期是所有硬件共享的时间单位 
  - 中断是时间线上发生的控制流切换 
     

 如果你把这 4 句话想明白，后面的 PPU、DMA、Mapper IRQ 都会顺很多。
