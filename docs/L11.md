第 11 课：栈、PHA/PLA、JSR/RTS 为什么最容易写错

 你现在已经能理解：

  - CPU 有 SP
  - 栈在 0x0100..0x01FF
  - SP 只有 8 位
  - JSR/RTS 和一些中断流程都依赖栈

 这节课的目标是把这套东西真正讲顺。
 如果这节你听明白了，后面实现 PHA/PLA/JSR/RTS/BRK/RTI 会稳很多。

---------

  1. 什么是栈

 先用程序员最熟悉的方式说： 
 栈 = 后进先出（LIFO）的数据结构 
 也就是： 

  - 最后压进去的 
  - 最先弹出来 

 这和很多语言里的函数调用栈一个思路。 
 在 CPU 里，栈通常用来做这些事： 

  - 暂存寄存器值 
  - 保存返回地址 
  - 保存状态寄存器 
  - 支持子程序调用和中断 

 所以栈不是可选小工具，而是 CPU 控制流的一部分。 

---------

2. 6502 的栈为什么特别 

 6502 的栈不是随便放在哪都行。 
 它固定在一整页内存里： 

  - 0x0100..0x01FF 

 这页叫 stack page。 
 所以： 

  - SP 只有 8 位 
  - 高 8 位永远固定是 0x01 

 也就是说： 

 实际栈地址 = 0x0100 + SP 

 这就是你之前答对的那个点。 

 所以如果： 

  - SP = 0xFD 

 那当前栈操作涉及的位置就是： 
  - 0x01FD 

---------

3. 栈是向上长还是向下长 

 6502 栈是向下长的。 
 这很重要。
 所谓“向下长”意思是： 

  - 压栈时，先写当前 SP 对应的位置 
  - 然后 SP -= 1 
  - 弹栈时，先 SP += 1 
  - 再读那个位置 


 所以先记这两条： 

```rust
  push                                               
  write(0x0100 + SP, value)
  SP -= 1
  pop                                                        
  SP += 1
  read(0x0100 + SP)
```

 很多人第一次写反，就是因为没适应“向下长”。 

---------

  4. 为什么压栈和弹栈顺序这么容易写错 
     

 因为它不是： 

  - 先减再写 
  - 或先读再加 
    

 而是有固定顺序。 

 举个例子： 

 假设现在： 

  - SP = 0xFD 
    

 执行一次 push： 

  1. 写入 0x01FD 
  2. SP 变成 0xFC 
     

 再执行一次 push： 

  1. 写入 0x01FC 
  2. SP 变成 0xFB 
     

 所以你看，SP 总是指向“下一个可写槽位之下”。 

 这也是为什么很多人会在脑子里糊掉。 

---------

  5. Rust 里你该先写哪两个函数 
     

 在做 JSR/RTS/PHA/PLA 之前，先把这两个辅助函数写对： 

```rust
fn stack_push(&mut self, value: u8, bus: &mut impl CPUBus) { 
    let addr = 0x0100 | self.sp as u16; 
    bus.cpu_write(addr, value); 
    self.sp = self.sp.wrapping_sub(1); 
} 

fn stack_pop(&mut self, bus: &mut impl CPUBus) -> u8 { 
    self.sp = self.sp.wrapping_add(1); 
    let addr = 0x0100 | self.sp as u16; 
    bus.cpu_read(addr) 
} 
```

 这两个函数只要对了，后面会轻松很多。 

 你现在最重要的是： 

 先把栈抽成独立辅助逻辑，不要在 JSR/PLA/RTI 里每次手写 

---------

  6. PHA 和 PLA 是最适合拿来练手的栈指令 
     

 在上 JSR/RTS 之前，最适合先做： 

  - PHA 
  - PLA 
    

 因为它们只涉及栈，不涉及复杂返回地址计算。 

 PHA 

  - 把 A 压栈 
    

```rust
fn pha(&mut self, bus: &mut impl CPUBus) {
    self.stack_push(self.a, bus); 
} 
```

 PLA 

  - 从栈弹一个字节到 A 
  - 更新 Z/N 
    

```rust
fn pla(&mut self, bus: &mut impl CPUBus) { 
    self.a = self.stack_pop(bus); 
    self.set_zn(self.a); 
} 
```

 注意： 

  - PHA 不改 Z/N 
  - PLA 会改 Z/N 
    

 这又回到了我们前面说的那条规则： 

 flag 是否更新，取决于指令定义，不取决于“有没有改寄存器”这种直觉 

---------

  7. JSR 到底做了什么 
     

 JSR = Jump to SubRoutine 

 也就是“调用子程序”。 

 你可以把它想成函数调用，但一定不要直接把它当高级语言函数，因为它有很明确的硬件行为。

 JSR target 的核心动作是： 

  1. 把“返回地址”压栈 
  2. 把 PC 改成 target 
     

 注意重点： 

 压栈的不是 target，而是返回地址 

 也就是说，CPU 要记住： 

 “这个子程序结束后，我应该回哪儿继续执行？” 

---------

  8. 为什么 JSR 最容易出 off-by-one 错误 
     

 因为“压哪个地址”很容易搞错。 

 6502 的 JSR 不是简单地： 

  - 压当前 PC 
  - 再跳走 
    

 而是有它自己的细节规则。 
 你如果直接按“函数调用直觉”写，十有八九会偏 1。 

 先用你当前阶段够用的理解来讲： 

  - JSR 读取 16 位目标地址 
  - 它会把“下一步恢复执行所需的地址信息”压栈 
  - 然后跳到目标地址 
    

 而 RTS 弹出来后还会再做一次地址调整 

 所以你会发现： 

 JSR 和 RTS 不是各自独立好写的，它们必须成对理解` 

---------

  9. RTS 到底做了什么 
     

 RTS = ReTurn from Subroutine 

 它的核心动作： 

  1. 从栈里弹出返回地址 
  2. 对地址做 6502 规定的调整 
  3. 把它写回 PC 
     

 这里的“调整”正是很多 off-by-one bug 的来源。 

 所以我建议你不要靠记忆碎片写 JSR/RTS，而是永远成对思考： 

  - JSR 压了什么 
  - RTS 取了什么 
  - 为什么 RTS 之后会回到调用点后面的那条指令 
    
---------

  10. 一个高层类比 
      

 假设程序是这样： 

 0x8000: JSR \$9000 
 0x8003: LDA #\$01 
 ... 
 0x9000: ... 
 0x9005: RTS 

 你真正想要的效果是：

  - 执行 JSR 
  - 跳到 0x9000 
  - 子程序跑完 
  - RTS 
  - 回到 0x8003 
    

 也就是说： 

 JSR/RTS 共同维护的是“回来以后从哪接着跑”` 

 这就是栈保存返回地址的意义。 

---------

  11. 为什么中断也依赖栈 
      

 虽然这节先不讲完整中断流程，但你现在应该先有这个认识： 

 中断发生时，CPU 也得暂时离开当前执行流。 
 那它之后怎么回来？ 

 当然也得保存现场。 

 这通常包括： 

  - 当前 PC 
  - 当前状态寄存器 
    

 所以中断同样依赖栈。 

 这也说明： 

 栈不是只给子程序用的，它是 CPU 控制流切换的基础设施 

---------

  12. 你现在实现时最容易犯的错 
      

 我提前帮你排雷。 

 错 1：push/pop 顺序反了 

  - 先减后写 / 先读后加 写反 
    

 错 2：PLA 忘了更新 Z/N 

  - 很常见 
    

 错 3：PHA 错误地更新了 flag 

  - 不该更新 
    

 错 4：JSR 压错地址 

  - 最常见的 off-by-one 来源 
    

 错 5：RTS 弹出地址后不做正确调整 

  - 导致返回到错误位置 
    

 所以这节课后你先别急着上 BRK/RTI，先把： 

  - stack_push 
  - stack_pop 
  - PHA 
  - PLA 
  - JSR 
  - RTS 
    

 这一组拿稳。 

---------

  13. 你现在最合理的实现顺序
      

 建议严格按这个顺序： 

  1. stack_push 
  2. stack_pop 
  3. PHA 
  4. PLA 
  5. 给 PHA/PLA 写测试 
  6. JSR 
  7. RTS 
  8. 给 JSR/RTS 写配对测试 
     

 不要跳步。 

 先把“字节压栈/弹栈”确认对，再碰“返回地址压栈/弹栈”。 

---------

  14. 这节课你必须记住的 6 句话 
      
  1. 6502 的栈固定在 0x0100..0x01FF。 
  2. SP 只有 8 位，因为高位固定是 0x01。 
  3. 6502 栈向下长：push 后 SP--，pop 前 SP++。 
  4. PHA 压 A，PLA 弹到 A 并更新 Z/N。 
  5. JSR/RTS 的本质是通过栈保存和恢复返回地址。 
  6. JSR/RTS 最容易出错的地方是返回地址的 off-by-one 处理。
