第27课：PPU 地址空间、\$2006/$2007 和读缓冲 

你其实已经提前实现了一部分，所以这节课重点不是“抄代码”，而是把你已经写出来的东西真正讲明白。 

先给核心结论： 

  - CPU 不能直接访问 PPU 内部显存 
  - CPU 先通过 $2006 设置 PPU 地址 
  - 再通过 $2007 读/写这个地址上的数据 
  - 而且 读 $2007 不是总能立刻读到真实值，它有一个非常重要的 读缓冲机制 

 这就是为什么 PPU 访问比普通 RAM 更像“协议”，而不像普通数组。 

---------

    1. PPU 自己也有地址空间 

你现在已经知道 CPU 地址空间和 PPU 地址空间不是一回事。 

PPU 内部常见的地址范围可以先记成： 

  - 0x0000..=0x1FFF：Pattern Table 
  - 0x2000..=0x2FFF：Name Table 
  - 0x3000..=0x3EFF：前面 Name Table 的镜像 
  - 0x3F00..=0x3FFF：Palette 

 你在 D:/code/rut/nes_core/src/bus.rs:98 已经把这套最小映射搭起来了，这很好。 
 
---------

    2. $2006 干的是“设置 PPU 当前地址” 

 $2006 不是普通寄存器，它是“地址输入口”。 
 CPU 写它两次： 

  1. 第一次写高字节 
  2. 第二次写低字节 
      

然后 PPU 内部才真正得到一个完整地址。 

这正是你在 D:/code/rust/nes_core/src/ppu/mod.rs:187 做的事。 

所以你现在脑子里要固定一个模型：

  - $2006 不是“值本身重要” 
  - 它的重要性在于：它改变了 $2007 将要访问哪里 
     
---------

    3. $2007 干的是“访问当前 PPU 地址上的数据” 

 当 $2006 已经设置好内部地址后： 
  - 写 $2007：向当前 vram_addr 写数据 
  - 读 $2007：从当前 vram_addr 读数据 
     

 然后地址还会自动递增。 
 这一步你在 D:/code/rust/nes_core/src/ppu/mod.rs:168 和 D:/code/rust/nes_core/src/ppu/mod.rs:198 已经接上了。
 
---------

  4. 为什么地址会自动递增 
      

 这是 PPU 协议的一部分。 
 你通过 \$2007 连续读写时，不需要每次都重新写 \$2006，因为访问后地址会自动前进。 

 而前进多少，由 $2000 的某一位决定： 

  - +1 
  - 或 +32 
     

 这就是你定义的 CTRL_VRAM_INCREMENT，实现见 D:/code/rust/nes_core/src/ppu/mod.rs:198。 

 你可以这样理解： 

  - +1 更像横向连续访问 
  - +32 更像按 Name Table 行步进 
---------

  5. 最重要的难点：为什么读 $2007 有缓冲 
      

 这是 PPU 最容易让初学者困惑的地方。 

 对于大部分 PPU 地址： 

  - 第一次读 $2007，你拿到的不是当前地址真实数据 
  - 而是“上一次缓冲里的旧值” 
  - 同时 PPU 再把当前地址真实数据装进缓冲 
  - 下一次读，才读到上一次装进来的值 

 这就是 延迟一拍的读缓冲。 
 
 你在 D:/code/rust/nes_core/src/ppu/mod.rs:152 已经按这个模型写出来了。 
 
---------

  6. 为什么会有这个读缓冲 
      

 你当前阶段不用去追电路细节，只要记住： 
 
  - PPU 不是普通 RAM 
  - CPU 访问它是通过寄存器窗口进行的 
  - 这个窗口读数据时有自己的协议延迟 
     

 所以读缓冲不是“奇怪的小技巧”，而是 PPU 协议本身的一部分。 
 
---------

    7. 为什么调色板区又是例外 

 Palette 区通常是： 
  - 0x3F00..=0x3FFF 

 这里读 $2007 时，行为和普通 VRAM 不完全一样： 
  - Palette 读通常会直接返回当前值 
  - 但内部缓冲仍然会按某种规则被更新 

 你在 D:/code/rust/nes_core/src/ppu/mod.rs:154 已经做了“palette 直返、同时刷新 read_buffer”的方向，这是对的。 

 这再一次说明： 

  - PPU 地址空间不是“所有区域规则都一样” 
  - 它是分区行为 
     
---------

  8. 为什么 \$2005/$2006 共用写锁存会这么麻烦 
      

 因为 PPU 不是每个寄存器都独立记完整状态。 
 有些寄存器写入是“第一下”和“第二下”含义不同。 

 所以你需要一个像 D:/code/rust/nes_core/src/ppu/mod.rs:22 里的 write_latch 这样的状态位。 

 这东西的意义是： 

  - 记录“下一次写入该被当成第几次写” 
     

而读 $2002 会把它清掉，见 D:/code/rust/nes_core/src/ppu/mod.rs:144。 

这就是为什么我前面一直强调： 
 

  - PPU 寄存器是协议状态机 
  - 不是几个独立的普通变量

  10. 这节课最关键的结论

  - $2006 负责设置 PPU 内部地址
  - $2007 负责通过这个地址窗口读写数据
  - 地址会自动递增
  - 大多数区域读 $2007 有延迟缓冲
  - Palette 区是特殊行为
  - 这说明 PPU 内存访问是“协议式访问”，不是普通 RAM 访问
