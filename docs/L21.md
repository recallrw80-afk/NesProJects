第21课：DMA 该挂在哪，Bus / CPU / PPU 到底谁负责 

 先给结论： 

  - 触发入口 在 Bus 
  - 执行暂停 体现在 CPU 
  - 写入目标 是 PPU/OAM 
  - 整体协调者 仍然应该是系统级总线/主机，而不是让某一个单独模块全包 
     

 也就是说，DMA 不是只属于 CPU，也不是只属于 PPU。 
 它本质上是“跨设备的数据搬运事件”，所以最自然的归属点是 Bus/整机层。 

---------

  1. 为什么不该把 DMA 只写进 CPU 
      

 如果你把 DMA 完全写死在 CPU 里，就会出现一个问题： 
 
  - CPU 不应该知道 OAM 在哪 
  - CPU 也不应该知道具体要向 PPU 内部哪个结构一字节一字节写 
     

 CPU 真正知道的只有： 
 
  - 有人往 $4014 写了一个页号 
  - 这会导致一段特殊总线行为 
  - 在这段期间 CPU 会暂停 
     

 所以 CPU 的职责更像： 
 
  - 接受“DMA 正在进行中” 
  - 在 DMA 完成前不要继续正常取指 
     

 而不是： 
 
  - 自己直接管理整段 OAM 内存拷贝细节 
     
---------

  2. 为什么也不该把 DMA 只写进 PPU 
      

 如果你把 DMA 完全塞进 PPU，也会不自然： 
 
  - 触发 DMA 的写地址是 CPU 地址空间里的 $4014 
  - 数据源来自 CPU 可见内存页 
  - CPU 还要被暂停 
     

 这些都不是 PPU 单方面能决定的。 
 
 所以 PPU 更像： 
 
  - DMA 的目标设备 
  - 提供“写 OAM 一字节”或“接收 OAM DMA 数据”的接口 
     

 而不是 DMA 总调度器。 

---------

  3. 最自然的职责划分 
      

 你可以这样理解： 
 
 Bus 
 
  - 负责识别写 $4014 
  - 记录 DMA 请求，例如“源页 = 0x02” 
  - 持有 DMA 状态，例如是否正在进行、当前搬到第几字节 
     

 CPU 
 
  - 每个时钟推进前先问一句： 
      - “总线现在是不是被 DMA 占用？” 
  - 如果是，就暂停正常取指 
  - 或者只参与 DMA 的读周期 
     

 PPU 
 
  - 提供 OAM 写入口 
  - 接收 DMA 搬来的 256 字节 
     

 这就是最清晰的模块边界。 
 
---------

  4. 你现在应该怎样把它想象成一条时间线 
      

 OAM DMA 不是一瞬间完成的，而是在时间线上展开： 
 
  1. CPU 向 $4014 写页号 
  2. Bus 记录“DMA 请求开始” 
  3. 接下来若干 CPU 周期里： 
      - DMA 从 CPU 地址空间读 1 字节 
      - DMA 写入 PPU OAM 1 字节 
  4. 共完成 256 次搬运 
  5. CPU 恢复正常执行 
      

 所以 DMA 更像一个“正在进行中的状态机”。 
 
 这点非常重要。 
 
---------

  5. 为什么它不适合写成一个普通函数 
      

 你可能会很自然地想写成： 

```rust
fn do_dma(page: u8) { 
    for i in 0..256 { 
        let data = bus.read(...); 
        ppu.write_oam(...); 
    } 
}  
```

 功能上这能跑，但从时序角度它是错的，或者至少会把你后面带偏。 

 因为这种写法把： 

  - 256 次读写 
  - CPU 暂停 
  - 周期消耗 
  - 与 PPU/APU 的同步关系 
     

 全部压扁成了“一个瞬间完成的函数”。 
 
 而真实硬件不是这样。 
 
 所以更合理的是： 
 
  - 记录 DMA 已经开始 
  - 每个 clock 推进一步 
  - 直到 256 字节传完 
     
---------

  6. 当前阶段推荐的数据结构思路 
      

 你不一定现在就写，但脑子里应该有这个模型： 

```rust
struct OamDma { 
    active: bool,
    page: u8, 
    offset: u8, 
    buffer: u8, 
    has_buffer: bool, 
}  
```

 你先别纠结字段长什么样，重点是理解： 

  - DMA 不是一个函数调用 
  - DMA 是一个持续多个时钟的过程 
  - 所以它天然适合“状态机” 
     
---------

  7. CPU 这边最需要暴露什么接口 
      

 CPU 不需要知道太多 DMA 细节。 
 它最需要知道的只有两类事：
 
  1. 现在能不能正常执行下一条指令 
  2. 当前这个 clock 是否被 DMA 占用 
      

 所以以后你可能会有类似这种交互： 

```rust
if bus.dma_active() { 
    bus.clock_dma(); 
    return; 
}  
```

 或者更系统一点： 

```rust
host.clock_cpu(); 
host.clock_ppu_three_times(); 
host.clock_dma_if_needed(); 
```

 具体形式以后再定，但核心思想就是： 

  - CPU 不主导 DMA 
  - CPU 只服从 DMA 对总线的占用结果 
     
---------

  8. 你现在要避免的一个架构误区 
      

 误区是： 
 
  - “谁被影响最大，就把逻辑全写在谁里面” 
     

 DMA 看起来最影响 CPU，于是很多人会把整个 DMA 都塞进 CPU。 
 这在前期可能省事，但后期会很难维护，因为： 
 
  - 触发地址是总线的事 
  - 数据源是总线映射的事 
  - 目标是 PPU 的事 
  - 暂停是 CPU 的事 
     

 所以 DMA 天生就是跨模块逻辑。 
 跨模块逻辑最适合放在整机协调层，而不是某个单设备里。 
 
---------

  9. 你当前阶段的最优实现目标 
      

 现在不用追求一上来就完全逐周期。 
 你可以分两步： 
 
 第一步：功能正确版 
 
  - 写 $4014 后触发 DMA 
  - CPU 暂停一段时间 
  - 256 字节正确搬到 OAM 
     

 第二步：时序更正确版 
 
  - DMA 按每拍推进 
  - 区分读拍/写拍 
  - 处理奇偶周期差异 
  - 和 PPU 同步推进 
     

 你现在处于适合先做第一步、理解第二步的阶段。 
 
---------

  10. 这节课你要带走的一句话 
       
  - DMA 属于整机总线时间线，不属于某一个孤立模块。
