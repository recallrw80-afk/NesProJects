第20课，DMA 是什么，为什么它不像 memcpy，而像“硬件抢总线”

 先给核心结论：

  - DMA 不是普通函数调用
  - DMA 也不是中断
  - DMA 是某个硬件要求“我现在要连续搬一大块数据，CPU 先让开总线”

 在 NES 里，你当前最重要的是 OAM DMA：

  - CPU 向 $4014 写入一个字节
  - 这个字节不是“数据本身”，而是一个 页号
  - 然后硬件会从 CPU 地址空间里的
    XX00 ~ XXFF
    连续读取 256 字节
  - 再把这 256 字节写进 PPU 的 OAM（精灵内存）

 比如：

  - 写入 $4014 = 0x02
  - 就表示从 0x0200 ~ 0x02FF 搬 256 字节

 所以 $4014 更像一句命令：

  - “把这一页拷贝到 OAM 去”

 为什么这不是普通 memcpy

 因为普通 memcpy 是 CPU 自己一条条指令搬。
 OAM DMA 则是：

  - 你只写一次 $4014
  - 后面的 256 字节传输由硬件机制完成 
  - CPU 在这段时间里基本上会被“卡住” 
     

 这就是“DMA 抢总线”的含义： 
 
  - 总线本来给 CPU 用
  - 现在 DMA 传输占用了它 
  - CPU 暂时不能正常取指执行 
     

 为什么它和中断不同 

 中断是： 

  - 保存现场 
  - 跳去执行一段代码 
  - 再回来 
     

 DMA 是： 
 
  - 不跳去执行程序 
  - 不运行中断处理函数 
  - 只是硬件层面强行占用总线做搬运 
     

 所以你可以这样记： 
 
  - 中断：改变控制流 
  - DMA：改变总线使用权 
     

 你当前阶段最该理解的 4 点 
 
  1. $4014 写入的是“源页号”，不是 256 字节数据本身。 
  2. OAM DMA 固定搬 256 字节。 
  3. DMA 期间 CPU 不是“去执行别的代码”，而是基本停住。 
  4. 这件事之所以难，是因为它发生在“时间轴”上，不是一个普通函数调用。 
      

 课后问题 
 
  1. 为什么 OAM DMA 更像“硬件搬运命令”，而不是一段 CPU 程序？ 
  2. 向 $4014`` 写入 0x03` 后，DMA 会从哪段地址读数据？ 
  3. 为什么说 DMA 的本质是“抢总线”，而不是“触发中断”？ 
  4. 如果 CPU 和 DMA 同时都想访问总线，为什么必须有一方暂停？ 
      

 你先回答这 4 个问题。答完后，我下一条就继续按你现在的 Rust 架构，讲 DMA 应该挂在 Bus、CPU 还是独立控制器上。 

---

1. OAM DMA是由其他设备触发的连续数据块搬运，不是CPU指令。
2. 向$4014写入0x03触发DMA，会读取0x0300-0x03FF连续256字节的数据写入到OAM。 
3. 中断通常会涉及跳转、执行一段任务然后跳转回来继续执行，而DMA则是强行打断CPU的工作转而执行数据帮运，这不是执行指令，不会涉及跳转。
4. DMA会占用总线的使用权，在期间CPU需要暂时停止对总线的访问，否则CPU继续执行指令会干扰数据的搬运。
