这节课的目标

你要真正听明白 4 件事：

---------

一、先建立正确直觉：PPU 不是按 tile 渲染，而是按像素时钟渲染

你可以先忘掉“屏幕上是一块一块 tile”。

从硬件视角看，PPU 真正在做的是：

- 当前在第几条扫描线 scanline
- 当前在这一行的第几个点 dot/cycle
- 到了这个点，就要输出这个点的颜色

也就是说，PPU 的本质不是：

- “我要画第几个 tile”

而是：

- “现在轮到屏幕上的这个像素点了，我必须立刻给出颜色”

这就是为什么它必须提前准备数据，而不能等到要画这个像素时再慢慢查。

---------

二、背景像素是怎么来的

背景一个像素的颜色，不是直接存在 Name Table 里的。

它要经过 4 层信息组合：

1. Name Table 给你 tile id
2. Attribute Table 给你这个区域用哪个调色板组
3. Pattern Table 给你这个 tile 当前这一行的两组 bitplane
4. 再结合 fine_x 取出当前像素对应的那 1 个 bit

所以一个背景像素，实际上是：

- tile 像素值，范围 0~3
- palette 编号，范围 0~3
- 两者拼起来去 palette 区取颜色

三、为什么是“每 8 个 dot 一组”

因为一个 tile 宽 8 像素。

PPU 在画背景时，逻辑上就是：

- 先把“下一个 tile 的数据”取回来
- 装进 shifter
- 然后接下来 8 个 dot，每个 dot 左移一次，从 shifter 顶部吐出一个像素

所以你可以把背景流水线理解成：

- 后台在取“下一个 tile”
- 前台在吐“当前 tile 的 8 个像素”

这两个动作是并行重叠的。

这就是你现在 fetch_bg() 和 update_bg_shifters() 同时存在的原因。

四、你这版 fetch_bg 的 8-dot 节奏

你代码里这一段很关键：

它按 self.cycles & 0x07 分 8 个阶段：

- 0：装载上一个 tile 的 shifter，同时读取新的 NT
- 2：读取 AT
- 4：读取 pattern low
- 6：读取 pattern high
- 7：increment_x()

你先不要死背具体数字，先记住这个思想：

每 8 个 dot，PPU 要把下一个 tile 的 4 份背景信息准备好。

这 4 份是：

- tile 编号
- attribute
- low plane
- high plane

五、shifter 到底是什么

shifter 可以把它想成“正在输送当前像素流的传送带”。

你现在有 4 个背景 shifter：

- bg_pattern_shift_lo
- bg_pattern_shift_hi
- bg_attr_shift_lo
- bg_attr_shift_hi

作用分别是：

- 前两个负责吐出背景像素的 2 bit
- 后两个负责吐出调色板编号的 2 bit

每个 dot 都左移一位：

六、为什么 attribute 也要做成 shifter

很多初学者会觉得：

- pattern 要 shift，我理解
- 颜色编号不是一整个 tile 都一样吗，为什么也 shift？

原因是硬件想要统一：

- 每个 dot 输出像素时，直接同时从 pattern shifter 和 attr shifter 取位
- 不要额外再做一次“当前像素属于哪个 palette”的判断

所以你现在把 attribute 的 2 bit 展开成 0x00 或 0xFF 装进 shifter，这是很常见也很实用的做法：

D:/code/rust/nes_core/src/ppu/mod.rs:466

这相当于把“这个 tile 的 palette 编号”复制成 8 个相同的 bit，和 pattern 一起同步输出。

七、loopy_v / loopy_t / fine_x 到底是什么

这是 PPU 背景滚动的核心。

你可以先这样理解：

- fine_x：当前 tile 内部的横向偏移
- loopy_v：当前正在渲染/访问的 VRAM 地址
- loopy_t：CPU 通过 \$2005/$2006 准备好的临时滚动地址

也就是说：

- CPU 改滚动时，不是立刻乱改当前渲染地址
- 而是先写进 t
- 到了硬件规定的时机，再把 t 的一部分拷进 v

这就是为什么会有：

- transfer_x()
- transfer_y()

八、为什么要分 increment_x / increment_y

因为背景滚动不是普通二维坐标。

PPU 的背景地址里混着这些信息：

- coarse X
- coarse Y
- nametable 选择位
- fine Y

所以横向走一个 tile，和纵向走一行像素，它们的更新规则完全不一样。

这就是为什么你必须单独写：

- increment_x()
- increment_y()

对应代码：

D:/code/rust/nes_core/src/ppu/mod.rs:501
D:/code/rust/nes_core/src/ppu/mod.rs:516

你现在这部分思路是对的：

- increment_x() 处理 coarse X 到 31 后回绕，并切换水平 nametable
- increment_y() 处理 fine Y 到头后进位到 coarse Y，并在特定位置切换垂直 nametable

这不是“代码写法问题”，这是 PPU 硬件协议本身就这么奇怪。

---------

九、为什么 transfer_x / transfer_y 不能随便做

这一步最容易在概念上混。

很多人会以为：

- CPU 既然已经写了滚动，那 PPU 直接一直用新值不就好了？

不行。

因为 PPU 正在一边渲染，一边取背景数据。

如果你在错误的时机把滚动值整体切过去，就会让本行后半段背景取图突然错乱。

所以硬件规定：

- 某个时机只复制 X 相关位
- 某个时机只复制 Y 相关位

你代码里也已经体现了这种“按时机拷贝部分位”的思想：

D:/code/rust/nes_core/src/ppu/mod.rs:545
D:/code/rust/nes_core/src/ppu/mod.rs:554

这说明你不是在“模拟一个方便理解的软件滚动变量”，而是在模拟 PPU 真正的地址流水线。

这是很大的进步。

十、把你的 clock() 用人话翻译一遍

你现在的主流程大概可以翻译成：

1. 如果到了 VBlank 开始点，置位 VBlank
2. 如果到了 pre-render 开始点，清除 VBlank/sprite flags
3. 如果当前是渲染行并且开启渲染：
4. 可见像素区先输出当前像素
5. 每个取图周期推进背景 shifter
6. 每 8 个 dot 取下一块背景数据
7. 行尾处理 increment_y
8. 某个时机拷回 X 滚动
9. pre-render 某段时机拷回 Y 滚动
10. 周期加一，扫描线推进，最后进入下一帧

这已经是“像 PPU”了，不再只是“有几个寄存器的模块”。

---------

十一、你现阶段最该抓住的结论

你只需要牢牢记住这 3 句：

1. 背景渲染的本质是“每个 dot 输出一个像素”，不是“每次画一个 tile”。
2. shifter 是为了把 tile 数据变成连续的像素流。
3. v/t/x 和 increment/transfer 是为了让滚动在渲染流水线里按硬件规则生效。

如果这三句你真的懂了，后面 sprite、sprite 0 hit、滚动分屏、扫描线时序，你都会容易很多。
