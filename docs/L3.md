第 3 课：先建立 NES 的最小系统图 这节课不讲具体指令，也不讲寄存器细节。 目标只有一个： 

 把 NES 看成一组互相协作的模块，并知道它们的数据流和控制流怎么走。 如果第 1、2 课讲的是“名词解释”，第 3 课讲的是“这些名词怎么连起来”。 

---------

1. 先看最小系统图 

 你先记住这个版本： 
 ```
               +------------------+
               |    Cartridge     |    
               | PRG / CHR / Map  |    
               +--------+---------+    
                        |             
                        v             
      +--------+    +--------+    +--------+   
      | Input  | -> |  Bus   | <- |  CPU   |   
      +--------+    +--------+    +--------+   
                        |              |       
                        |              v        
                        |         game logic   
                        |           
               +--------+---------+    
               |                  |    
               v                  v
            +------+          +------+   
            | PPU  |          | APU  | 
            +------+          +------+   
               |                  |    
               v                  v     
            Video              Audio
 ```

 你现在先不要在意这个图是不是“电路级精确”，它的目的是帮你建立程序结构直觉。 这个图里的核心关系是： 

  - CPU 通过 Bus 访问整个系统 
  - Cartridge 给 CPU 和 PPU 提供游戏数据 
  - PPU 输出视频 
  - APU 输出音频 
  - Input 把玩家输入变成寄存器状态供 CPU 读取 

---------

2. 谁在主动做事，谁在被动响应 

 这是理解模拟器结构的关键。 主动推进的模块 
  - CPU 
  - PPU 
  - APU 
  - DMA 

 它们会随着时钟一步一步前进。 被动响应的模块 
  - Bus 
  - Cartridge 
  - Mapper 
  - Input 寄存器 

 它们通常是在“有人访问它们时”才响应。 这意味着： Bus 一般不是“自己跑”的模块，它更像一个调度入口。 真正不停变化的是 CPU/PPU/APU 这些时序驱动模块。 

---------

3. CPU 在系统里的地位 

 CPU 是整个系统的“主控制者”，但不是“唯一工作者”。 CPU 负责： 
  - 从程序 ROM 里取指令 
  - 读写地址空间 
  - 修改 RAM、寄存器、PPU/APU/手柄相关状态 
  - 响应中断 

 但 CPU 不负责： 
  - 自己画图 
  - 自己发声 
  - 自己决定卡带如何切页 

 所以你可以说： CPU 负责运行游戏逻辑，但不负责亲自完成所有硬件工作。 举例： 
  - 游戏代码要滚动画面，CPU 会写 PPU 寄存器 
  - 游戏代码要播放声音，CPU 会写 APU 寄存器 
  - 游戏代码要读按键，CPU 会读手柄寄存器 

 也就是说，CPU 很像“发命令的人”。 

---------

4. PPU 为什么经常比 CPU 更难 

 很多初学者以为： “CPU 最核心，所以肯定最难。” 实际上，入门阶段 CPU 更抽象，PPU 更容易让人崩溃。 原因是： 
  - CPU 是“顺序执行”的，逻辑线性 
  - PPU 是“按像素、按扫描线、按帧”推进的，时序复杂
  - PPU 既有显存，又有滚动，又有调色板，又有精灵，又有背景混合 

 所以你现在要先有心理预期： CPU 是先打基础，PPU 是真正容易卡住的地方。 这也是为什么我们的课程顺序是先 CPU，再 PPU。 

---------

5. Cartridge 在系统里到底做什么 

 先忽略 Mapper，只看最基本角色。 Cartridge 主要提供两类数据： 
  - PRG：给 CPU 当程序/数据读 
  - CHR：给 PPU 当图块/图形数据读 

 你可以粗略理解成： 
  - CPU 看的“书”在 PRG 里 
  - PPU 画图用的“素材”在 CHR 里 

 所以 Cartridge 不是“只有一段 ROM”。 它更像是： struct Cartridge { prg_rom: Vec<u8>, chr_rom_or_ram: Vec<u8>, // 以后还有 mapper、mirroring、battery ram 等 } 

---------

6. Input 在 NES 里为什么不是普通事件系统 

 你平时做 GUI 时，可能习惯： 
  - 键盘事件回调 
  - 鼠标事件回调 

 但 NES 不是这套思路。 NES 里，手柄更像一个可读寄存器设备。 CPU 不是“等输入事件推送过来”，而是： 
  - 先写手柄寄存器让它锁存 
  - 再一位一位读出按键状态 

 这个项目里就在 0x4016/0x4017 处理了手柄：core/bus.c:61 所以你要先把“输入”从“事件回调思维”切换成“硬件寄存器思维”。 这对后面理解 joypad 很重要。 

---------

7. 一帧画面是怎么来的 

 先讲粗略版本，不讲像素级细节。 这个项目里，一帧的基本流程在 core/libretro.c:27： 
  1. 前端采集输入 
  2. 输入状态更新到总线/控制器寄存器 
  3. 核心不断推进 CPU、PPU、DMA 
  4. 直到 PPU 产出一整帧缓冲 
  5. 把帧缓冲交给前端显示 
  6. 把音频样本交给前端播放 

 你先把“一帧”理解成： 前端调用一次 run，核心就推进系统直到拿到下一张完整画面 所以这里有两个层次： 
  - 硬件层：CPU/PPU/APU 按周期推进 
  - 前端层：一帧一帧地拿结果显示 

 这两个层次不要混。 

---------

8. 前端和核心为什么一定要分开 

 因为这两者解决的问题不同。 核心 
  - NES 的硬件规则是什么 
  - 0x2002 该怎么读 
  - PPU 什么时候触发 NMI 
  - Mapper 怎么切 bank 

 前端 
  - 用 GLFW 还是 SDL 
  - 窗口多大 
  - 键盘映射哪个键 
  - 音频设备怎么打开 

 如果把它们混在一起，会发生两件坏事： 
  - 核心难测 
  - 平台相关逻辑污染硬件逻辑 

 所以对你用 Rust 重写来说，一个好的起点是： 
  - core 只暴露 tick()、frame_buffer()、push_input() 之类接口 
  - frontend 只负责把这些接口接到窗口/音频/按键 

---------

9. Rust 里第一版应该怎么建模 

 现在先不给你最复杂的所有权设计，只给最朴素的模型。 pub struct Nes { cpu: Cpu, ppu: Ppu, apu: Apu, bus: Bus, cart: Cartridge, } 这里你先只关心“模块存在”。 后面我们会发现一个 Rust 现实问题： 
  - CPU 需要通过 Bus 读写 
  - Bus 又要能访问 PPU、Cartridge、Input 
  - PPU 有时还要访问 Cartridge 的 CHR 

 这说明模块之间不是简单父子关系，而是“围绕总线互相协作”。 所以第 3 课你的任务不是立刻写最终架构， 而是先接受这个事实： NES 是一个模块互联的系统，不是一个单类对象里塞几百个函数。 

---------

10. 这节课最重要的理解 

 你现在要开始从“程序执行”视角，切到“系统协作”视角。 以前你可能会问： 
  - 这行代码先执行还是后执行？ 

 以后做模拟器你更常问的是： 
  - 这个地址归谁响应？ 
  - 这个寄存器是谁的？ 
  - 这个事件是谁触发的？ 
  - 这个时刻 CPU、PPU、DMA 谁在推进？ 
  - 这块数据到底属于主机 RAM、PPU VRAM 还是卡带 ROM？ 

 这就是为什么模拟器开发特别锻炼“系统建模能力”。 

---------

11. 这节课你必须记住的 5 句话 
      
  1. CPU 是主控制者，但不是唯一干活的模块。 
  2. PPU 负责图像，APU 负责音频，它们不是 CPU 的普通函数。 
  3. Cartridge 提供游戏程序和图形数据。
  4. Bus 把整个地址空间组织起来，让 CPU 能访问整台机器。 
  5. 模拟器开发的核心不是写算法，而是正确表达模块关系和时序关系。
