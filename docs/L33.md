第33课：Sprite Zero Hit 是什么，为什么它不是碰撞检测 

这节课很经典，也很容易被误解。

先给核心结论： 

- Sprite Zero Hit 不是游戏逻辑层的碰撞检测 
- 它是 PPU 的一个渲染状态事件 
- 它表示： 
- sprite 0 的某个非透明像素 
- 与背景的某个非透明像素 
- 在当前渲染过程中发生了重叠 


当这件事发生时，PPU 会把 PPUSTATUS 里的一个标志位置 1。 

这就是 Sprite Zero Hit。 

---------

1. 为什么叫 Sprite Zero 


因为 OAM 里一共有 64 个 sprite 条目，而： 

- 第 0 个 sprite 条目 
- 也就是 OAM 的第一个 sprite 


被特殊对待了。 

并不是所有 sprite 都能触发这个标志，只有： 

- sprite 0 


所以它叫 Sprite Zero Hit。 

---------

2. “Hit” 到底是什么意思 


这里的 “hit” 不是“角色撞墙了”那种游戏碰撞。 
它更准确地说是： 

- 在屏幕渲染时 
- sprite 0 的不透明像素 
- 和背景的不透明像素 
- 出现在同一个屏幕像素位置 


于是 PPU 记录下这个事件。 

所以它是： 

- 渲染层面的像素重叠事件 
- 不是物理碰撞 
- 不是游戏对象逻辑碰撞 

---------

3. 为什么它不是碰撞检测 


因为真正的游戏碰撞检测通常关心的是： 

- 角色和敌人是否相撞 
- 子弹是否打中目标 
- 矩形/圆形/像素区域是否接触 

而 Sprite Zero Hit 根本不关心这些游戏语义。 
它只关心一件很机械的事： 

- 当前画面输出时 
- sprite 0 的非透明像素有没有和背景非透明像素重叠 


它既不关心： 

- 这是不是玩家角色 
- 这是不是敌人 
- 这是不是应该扣血 


所以它绝不是通用碰撞系统。 

---------

4. 那它为什么这么重要 


因为很多 NES 游戏会利用它做： 

- 画面分割 
- 状态栏分离 
- 不同区域滚动效果切换 


也就是说，程序会故意把 sprite 0 放在某个特殊位置，然后等待： 

- “Sprite Zero Hit 发生了没有？” 


一旦发生，就说明： 

- PPU 已经渲染到屏幕上的某个位置了 


于是 CPU 就可以在那个时间点附近做一些寄存器修改，比如： 

- 改滚动 
- 改显示区域参数 


所以 Sprite Zero Hit 的真正价值是： 

- 给程序一个和屏幕扫描位置相关的时机信号 

---------

5. 为什么它能用来做画面分割 


因为它发生的条件和“当前扫描到哪里”有关。 

你可以粗略理解成： 

- 程序把 sprite 0 放在状态栏和主画面的分界附近 
- 当 PPU 扫到那里，sprite 0 和背景开始重叠 
- sprite zero hit 置位 
- CPU 轮询到这个标志后，就知道： 
- “扫描线已经到这里了” 
- 然后立刻修改 PPU 寄存器，改变后续区域的显示方式 


于是就能做出类似： 

- 上面是固定 HUD 
- 下面是滚动游戏画面 


这在 NES 上非常常见。 

---------

6. 触发它需要哪些条件 


当前阶段你先记最关键的几个： 

1. 必须是 sprite 0 
2. sprite 当前像素必须非透明 
3. 背景当前像素也必须非透明 
4. 两者必须出现在同一个屏幕像素位置 


只有同时满足这些条件，才会触发。 

所以不是说：

- “sprite 0 在这一行上”就一定 hit 
- 也不是说： 
- “sprite 0 和背景区域大致重叠”就一定 hit 


它是更细粒度的像素级条件。 

---------

7. 为什么透明像素不算 


因为透明像素本质上表示： 

- 这个像素位置不真的参与显示覆盖 


所以如果 sprite 0 当前那个像素是透明的，即使几何位置上重叠，也不算 hit。 

这再次说明： 

- Sprite Zero Hit 是渲染像素事件 
- 不是“边界框相交事件” 

---------

8. 这个标志放在哪里 


它在 PPUSTATUS (\$2002) 里。 
也就是说，CPU 可以通过读 $2002 来观察它。 

你现在不用急着背具体位号，但要知道： 

- $2002 不只包含 VBlank 
- 还包含 sprite 相关状态 
- sprite zero hit 就是其中之一 


你在当前代码里已经留了： 

- STATUS_SPRITE_ZERO_HIT


这很好，说明你已经在给它留位置了。 

---------

9. 这节课和你前面学的内容怎么连起来 


前面你已经学了： 

- sprite 来自 OAM 
- sprite 和背景最后要合成 
- PPU 是按扫描线/像素推进的 
- PPUSTATUS 是状态寄存器 


现在这些终于汇合成一个新概念： 

- 当 sprite 0 和背景在渲染时发生像素重叠 
- PPU 就在 PPUSTATUS 里记一笔 
- CPU 可以利用这件事判断“画面扫描大概到哪了” 


这就是 Sprite Zero Hit 的本质。 

---------

10. 当前阶段你先别做什么 


你现在先不要急着实现： 

- 精确 sprite zero hit 时机 
- 左边 8 像素裁剪的特殊规则 
- 各种边缘例外 


这些都属于更细的 PPU 精度问题。 

现在你最重要的是先把概念想清楚： 

- 它不是碰撞检测 
- 它是像素级渲染重叠事件 
- 游戏常把它当“扫描到某位置了”的信号用 

---------

11. 这节课的最简总结 

- Sprite Zero Hit 只和 sprite 0 有关 
- 它表示 sprite 0 非透明像素和背景非透明像素发生重叠 
- 它是渲染事件，不是游戏碰撞检测 
- 它常被用来做画面分割和状态栏效果 
- 这个状态记录在 PPUSTATUS
