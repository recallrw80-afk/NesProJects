第26课：先实现最小 \$2000/$2002，不渲染画面也把 PPU 的 NMI 骨架搭起来 

 这节课的目标不是画画，而是先让下面这条链路成立： 

  1. PPU 进入 VBlank 
  2. $2002 的 VBlank 位置 1 
  3. 如果 $2000.bit7 = 1，则触发 NMI 
  4. CPU 能收到这个 NMI 
  5. 读 $2002 会清掉 VBlank 标志 
      

 只要这条链成立，你的 PPU 就从“只有 OAM 的容器”升级成“真的会和 CPU 协作的设备”。 
 
---------

  1. 当前阶段你应该给 PPU 增加哪些最小状态 
      

 建议先只加这几个字段： 

```rust
pub struct PPU { 
    oam: [u8; 256], 
    oam_addr: u8, 
    ppuctrl: u8, 
    ppustatus: u8, 
    nmi_line: bool, 
} 
```
 你先别急着拆位字段，当前阶段 u8 就够了。 

 其中： 

  - ppuctrl：保存 $2000 
  - ppustatus：保存 $2002 
  - nmi_line：告诉外部“PPU 当前是否在拉起 NMI 请求” 

---------

    2. $2000 最小只关心什么 

 当前你最该关心的只有一位： 
  - bit7：VBlank 时是否输出 NMI 

 所以你可以先写一个 helper： 

```rust
fn vblank_nmi_enabled(&self) -> bool { 
    (self.ppuctrl & 0x80) != 0 
} 
```

 这样后面逻辑会很清晰。 
 

---------

    3. $2002 最小只关心什么 

 当前你最该关心的也先只有一位： 
  - bit7：VBlank 标志 

 先写两个 helper： 

```rust
fn set_vblank(&mut self, value: bool) 
fn in_vblank(&self) -> bool  
```

 比如： 

```rust
fn set_vblank(&mut self, value: bool) { 
    if value { 
        self.ppustatus |= 0x80; 
    } else { 
        self.ppustatus &= !0x80; 
    } 
} 
```



---------

    4. CPU 读写寄存器时，最小行为该是什么 

 你当前 cpu_write_register() / cpu_read_register() 可以先扩成： 
  - 写 $2000：保存到 ppuctrl 
  - 读 $2002： 
      - 返回当前 ppustatus 
      - 清掉 VBlank 标志 
      - 关闭 nmi_line（至少在最小模型里可以这么做） 
         

 伪代码： 

```rust
pub fn cpu_write_register(&mut self, addr: u16, data: u8) { 
    match addr { 
        0x2000 => self.ppuctrl = data, 
        0x2003 => self.oam_addr = data, 
        0x2004 => self.write_oam_data(data), 
        _ => {} 
    } 
} 

pub fn cpu_read_register(&mut self, addr: u16) -> u8 {
    match addr { 
        0x2002 => { 
            let status = self.ppustatus; 
            self.set_vblank(false); 
            self.nmi_line = false; 
            status 
        } 
        0x2004 => self.oam[self.oam_addr as usize], 
        _ => 0, 
    } 
}  
```

 注意： 

  - 这里是“最小教学模型” 
  - 以后还要补 $2002 对写入锁存的影响，但现在先不做 
     
---------

    5. 那 PPU 什么时候进入 VBlank 

当前阶段先不要接真实扫描线时序。 

你可以先提供一个显式方法给测试或上层调用： 

```rust
pub fn enter_vblank(&mut self) { 
    self.set_vblank(true); 
    if self.vblank_nmi_enabled() { 
        self.nmi_line = true; 
    } 
} 

pub fn leave_vblank(&mut self) { 
    self.set_vblank(false); 
    self.nmi_line = false;
}  
```

 这一步非常重要，因为它把： 

  - “PPU 时间阶段变化” 
  - “状态位变化” 
  - “NMI 请求产生” 

 三件事连起来了。 
---------

    6. 为什么 enter_vblank() 很适合当前阶段 

 因为你现在还没有完整 PPU 时钟和扫描线。 

所以最合理的做法不是硬凹真实渲染，而是先手动制造这个事件： 

  - “现在 PPU 进入 VBlank 了” 

 这样你可以先把 CPU/PPU 协作逻辑搭好。 
 这就是“先做骨架，再补时序细节”的正确路线。 

---------

    7. Bus 和 CPU 怎么接这条 NMI 线

 你现在大概已经有 CPU 的： 
  - set_nmi(bool) 

 那接下来最自然的做法就是： 
  - Bus 或主机层在每次推进时把 ppu.nmi_line() 同步给 CPU 

例如概念上

cpu.set_nmi(ppu.nmi_line()); 

以后更完整时，会由系统时钟驱动器负责这件事。

---------

  9. 当前阶段最容易写错的地方 
     
     把“读 $2002 清标志”误写成“VBlank 时间真的结束” 
     这是错的，刚刚已经纠正过。 
     
     $2000.bit7 改了以后却不影响 NMI 逻辑 
     那就说明寄存器和行为没接上。 
     
     让 $2002 变成普通字段随便读，没有副作用
     这会丢掉 PPU 协议最重要的特征。

---------

  10. 这节课的真正目标

 不是实现完整 PPU，而是先让你拥有这个最小闭环：

  - CPU 写 $2000
  - PPU 进入 VBlank
  - PPU 根据 $2000 决定是否拉 NMI
  - CPU 读 $2002
  - VBlank 标志被清掉

 只要这条链成立，后面你再接扫描线时序、帧推进、真实 VBlank 时刻，就有落点了。
