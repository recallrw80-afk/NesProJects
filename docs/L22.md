第22课：为什么 OAM DMA 常常是 513 或 514 个 CPU 周期 

 这节是 DMA 最容易让人困惑的地方。 
 你前面已经知道： 

  - OAM DMA 会搬 256 字节 
  - 每字节不是“瞬间传送” 
  - 它发生在 CPU 时间轴上 
     

 现在问题来了： 
 
  - 256 字节，为什么不是 256 周期？ 
  - 为什么很多文档会写 513 或 514 周期？ 
     

 核心答案是： 
 
  - OAM DMA 不是“每字节 1 个周期” 
  - 而是基本上每字节要经历 读 + 写 两个阶段 
  - 再加上启动时机对齐问题，所以总数会变成 513 或 514 
     
---------

  1. 先建立最粗粒度直觉 
      

 搬 256 字节至少要做两类动作： 
 
  1. 从 CPU 地址空间读 1 字节 
  2. 向 PPU OAM 写 1 字节 
      

 所以直觉上就是： 

 256 次读 + 256 次写 = 512 个周期 

 这就是为什么答案一定不会是 256。 

---------

  2. 那多出来的 1 或 2 周期是什么 
      

 因为 DMA 不是任何时刻都能直接无缝开始。 
 它要对齐 CPU 的总线节拍。 

 可以先把它想成： 

  - DMA 启动时，可能先要等一个“准备拍” 
  - 然后才开始进入稳定的读/写交替节奏 
     

 于是就会出现： 
 
  - 某些情况下：512 + 1 = 513 
  - 某些情况下：512 + 2 = 514
     
---------

  3. 为什么会和奇偶周期有关 
      

 OAM DMA 的读写节奏依赖 CPU 时钟相位。 
 如果 DMA 触发时 CPU 正处于某个合适相位，它更快进入读写循环；如果不在合适相位，就要多等一拍。 

 你现在不用背硬件电路细节，只要记住： 

  - DMA 启动时机不同 
  - 对齐结果不同 
  - 所以总周期可能是 513 或 514 
     

 这本质上是一个“时序对齐问题”。 
 
---------

  4. 你可以用一个简化模型先理解 
      

 最简化版本可以这样想： 
 
  - 第 1 拍：DMA 准备/对齐 
  - 接下来 512 拍：256 次读 + 256 次写 
  - 如果启动点不对，再额外补 1 拍 
     

 所以得到： 
 
  - 513 周期
  - 或 514 周期 
     

 这已经足够支撑你当前阶段的设计。 

---------

  5. 为什么这件事说明 DMA 不是普通 memcpy 
      

 如果它只是个普通拷贝函数，你完全不会关心： 
 
  - 起始时钟相位 
  - 奇偶周期 
  - 是否要多等一拍 
     

 但硬件 DMA 必须关心这些，因为它不是“逻辑动作”，而是“时间线上占用总线的行为”。 
 
 这就是模拟器开发和普通业务开发最大的区别之一： 
 
  - 逻辑正确不够 
  - 时间也要正确 
     
---------

  6. 当前阶段你在代码里应不应该立刻实现 513/514 精确行为 
      

 我的建议是： 
 
  - 现在要理解 
  - 不一定现在就必须完全做对 
     

 你当前更重要的是先把架构摆正： 
 
  1. DMA 有独立状态 
  2. DMA 会阻塞 CPU 正常执行 
  3. DMA 逐时钟推进 
  4. 最终能正确搬完 256 字节 
      

 在这个基础上，再补“奇偶周期导致 513/514”的精度，成本会低很多。 
 
---------

  7. 如果你现在要做“第一版可用实现” 
      

 你可以先采取这样的策略： 
 
  - 记录 DMA 开始时的 CPU 时钟奇偶 
  - 给 DMA 一个剩余周期计数 
  - 按简化规则让它是 513 或 514 
  - 同时完成 256 字节搬运 
     

 哪怕你内部还没拆到“每拍读一次/写一次”，这也已经比“一次函数瞬间拷贝完”强很多。 
 
---------

  8. 真正更精确的实现会长什么样 
      

 以后更精确时，常见思路会是： 
 
  - DMA 有自己的 active/page/offset/data_latch 
  - 每个 CPU 周期只推进一步 
  - 某些拍读 CPU 内存 
  - 某些拍写 PPU OAM 
  - 启动时先根据当前 CPU 周期奇偶决定是否插入等待拍 
     

 这时 513/514 就不是“硬编码数字”，而是状态机自然跑出来的结果。 
 
 这是更高级的写法。 
 
---------

  9. 这一课你要真正记住的不是数字，而是原因 
      

 不要死背： 
 
  - “OAM DMA 是 513 或 514 周期” 
     

 你更应该记住： 
 
  - 因为它是 256 次读写交替 
  - 再加启动对齐等待 
  - 所以周期数取决于开始时刻 
     

 这才是对以后有用的理解。 
 
---------

  10. 这节课的最简总结 
       
  - 256 字节 DMA 不是 256 周期，而接近 256 * 2 
  - 读写交替带来 512 周期基础成本 
  - 启动对齐让它变成 513 或 514 
  - 这再次说明：DMA 是时间行为，不是普通数据复制函数
