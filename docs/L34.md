第34课：为什么一帧不是“画一张图”，而是扫描线和点逐步跑出来的 

这节课非常关键，因为从这里开始，你会真正把：

- PPU 
- VBlank 
- Sprite Zero Hit 
- NMI 
- 渲染时序 


全都放到一条统一时间线上去理解。 

先给核心结论： 

- NES 不是先在内存里画好一整张图，再一次性显示 
- 它更像一台持续工作的扫描输出设备 
- 一帧画面是在时间轴上，一点一点跑出来的 


所以“一帧”不是一个瞬间结果，而是一个完整的时序过程。 

---------

1. 什么叫扫描线 


你可以先把屏幕想成很多横向行。 

PPU 输出画面时，不是同时处理整张画面，而是： 

- 先处理第 0 行 
- 再第 1 行 
- 再第 2 行 
- … 
- 一行一行往下走 


每一行，就叫一条 scanline（扫描线）。 

所以： 

- 一帧 = 很多扫描线按顺序跑完 

---------

2. 什么叫 dot / cycle / 像素时刻 


即使在一条扫描线上，也不是整行一起完成。 
而是继续细分为： 

- 这一行上的第 1 个位置 
- 第 2 个位置 
- 第 3 个位置 
- … 


这些更细的小时间点，你现在可以先理解成： 

- dot 
- 或者“这一扫描线上的当前点位” 


所以 PPU 的时间推进是两层的： 

1. 当前在第几条扫描线 
2. 当前在这条线上的第几个点 


这就是为什么你代码里有： 

- scanline 
- dot 

它们不是装饰变量，而是 PPU **时间坐标**。 

- 它们和屏幕空间坐标有强关联，但不完全等同于“横纵像素坐标”

因为：

- 有些 scanline/dot 发生在不可见区域
- VBlank、预渲染线也占用这些时序位置
- 所以它们本质上是在描述“PPU 当前运行到时间线的哪里”，而不完全等同于当前画面渲染的位置。

---------

3. 为什么这比“画一张图”更难 


因为很多事件都不是“整帧结束才发生”，而是在某个具体扫描位置发生的。 

例如： 

- VBlank 在某个扫描线/点位开始 
- Sprite Zero Hit 在某个背景像素与 sprite 0 像素重叠的位置触发 
- 某些滚动/寄存器变化如果发生得太晚，就只影响后半部分画面 


所以 NES 画面不是“静态结果”，而是“时序生成过程”。 

---------

4. 你现在可以把一帧粗略分成三类阶段 


当前阶段先不死记细数字，先记阶段： 

1. 可见扫描线 

- PPU 真正在输出可见画面内容 

2. VBlank 阶段 

- 当前帧画完后的空白期 
- 这是 CPU 更新 PPU 相关状态的重要窗口 

3. 预渲染扫描线 

- 为下一帧开始做准备 
- 一些状态会在这里复位 


这样你先有大图景，后面再填数字。 

---------

5. 为什么 Sprite Zero Hit 必须放到这个模型里看 


如果你把屏幕理解成“整帧瞬间出现”，那 Sprite Zero Hit 很难理解。 

但一旦你接受： 

- PPU 正在逐点扫描输出 


那么 Sprite Zero Hit 就很好理解了： 

- 当扫描到某一个具体位置时 
- sprite 0 的非透明像素和背景非透明像素重叠 
- PPU 就在那个时间点把状态位置上 


所以 Sprite Zero Hit 本质上是： 

- 扫描过程中的时序事件 

---------

6. 为什么 VBlank 也必须放到这个模型里看 


VBlank 也不是一个抽象布尔值。 
它是： 

- “可见画面已经扫完” 
- “接下来进入空白阶段” 


所以 VBlank 也是时间结构的一部分。 

PPU 不是“随便哪一刻就 VBlank 了”，而是： 

- 跑到某个扫描线/点位 
- 进入 VBlank 
- 于是 $2002 的标志变化 
- 于是可能触发 NMI 


所以： 

- VBlank 是时序事件 
- NMI 是这个时序事件通知 CPU 的方式之一 

---------

7. 为什么你前面写的 clock() 模型是必要的 


如果 PPU 只是一个静态数据容器，那你根本不需要： 

- scanline 
- dot 
- clock() 


但你现在已经开始写这些了，原因就在于： 

- PPU 的很多行为依赖时间推进 
- 而不是只依赖“当前内存里有什么数据” 


这说明你已经开始从“数据结构思维”切到“硬件时序思维”。 

---------

8. 为什么 CPU 和 PPU 必须共用一条时间轴 


因为 CPU 不是在真空里运行。 

你前面已经学过： 

- 大致上 1 次 CPU 对应 3 次 PPU


这意味着： 

- CPU 做一件事时，PPU 也在继续往前扫 
- 所以 CPU 改寄存器的时机，会决定到底影响哪一部分画面 


这就是为什么： 

- 同样一段代码 
- 在不同扫描时刻执行
- 可能产生不同显示结果 


所以整机模拟必须把 CPU 和 PPU 放在同一条时间轴上理解。 

---------

9. 你现在最该建立的脑图


不要再想： 

- “一帧就是一个 framebuffer” 


而要想： 

master clock 持续推进 
PPU 在 scanline / dot 上前进 
CPU 以更慢速度前进 
PPU 在特定时刻进入 VBlank 
PPU 在特定时刻可能触发 sprite zero hit 
CPU 在这些时刻前后修改寄存器，会影响之后扫描到的画面 

这才是 NES 的真实味道。 

---------

10. 当前阶段你不需要先背的数字 


很多资料会直接告诉你： 

- 262 条扫描线 
- 每条 341 个点 
- 哪几条是可见区 
- 哪几条是 VBlank 


这些当然重要，但你现在更重要的是先把“为什么必须这样思考”搞明白。 

数字后面随时能补，思维模型错了最麻烦。 

---------

11. 这节课的最简总结 

- 一帧不是瞬间生成的图，而是扫描过程 
- PPU 一行一行、一点一点输出画面 
- scanline 和 dot 就是 PPU 的时间坐标 
- VBlank、Sprite Zero Hit、NMI 都是这条时间轴上的事件 
- CPU/PPU 必须放在同一条时间线上理解
