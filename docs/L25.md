第25课：VBlank 和 NMI，PPU 为什么要在某个时刻通知 CPU 

 这节课是你第一次把： 

  - PPU 寄存器 
  - PPU 时序 
  - CPU 中断 

 三件事真正连起来。 
 先给核心结论： 

  - VBlank = 垂直空白期 
  - 它是 PPU 在一帧画面扫完之后、下一帧开始之前的一小段“安全窗口” 
  - 在这段时间里，CPU 更适合去更新 PPU 相关数据 
  - 如果开启了 PPUCTRL 的 NMI 开关，PPU 会在 VBlank 开始时触发一次 NMI 

 这就是你前面学的 NMI 在 NES 里的最经典来源。 

---------

1. 为什么会有 VBlank 

 你先不要想 LCD 屏幕，先按老式扫描显示设备的直觉来理解： 
  - PPU 不是“整张图一瞬间生成” 
  - 它是在时间轴上，一行一行把画面扫出来 

也就是说，一帧画面不是静止的内存块，而是持续输出过程。 

那就会有两个阶段： 

  1. 正在显示画面 
  2. 这一帧显示完了，下一帧还没开始 

 第二个阶段，就是 VBlank。 

 所以 VBlank 的本质是： 

  - 当前帧显示结束后的空档期 

---------

2. 为什么这段时间特别重要 

因为当 PPU 正在输出画面时，如果 CPU 这时乱改某些 PPU 数据，画面可能会出问题。 

你可以先粗略理解成： 

  - PPU 正在忙着“把当前这一帧画出去” 
  - 这时 CPU 不适合随便动背景/滚动/显存相关数据 
  - 等到 VBlank 来了，说明： 
      - 这一帧已经画完 
      - 现在有一小段较安全的时间可更新下一帧要用的数据  

 所以 VBlank 对游戏程序来说，最重要的意义就是： 
  - 通知 CPU：现在是更新画面状态的好时机 

---------

3. 为什么 PPU 要用 NMI 来通知 CPU 

 因为 CPU 不会天然知道： 
  - PPU 什么时候画完了一帧 

最笨的方法当然可以是 CPU 一直轮询，但这很低效。 

于是 NES 提供了更好的机制： 

  - 如果 $2000 里启用了“VBlank 触发 NMI” 
  - 那么 PPU 一进入 VBlank 
  - 就立刻给 CPU 一个 NMI 

 于是 CPU 会： 
  1. 暂停当前主线 
  2. 进入 NMI 处理程序 
  3. 在里面做本帧更新 
  4. RTI 回去 

 所以从游戏程序视角看，NMI 常常就意味着： 
  - “一帧结束了，开始准备下一帧吧” 

---------

4. $2002 和 VBlank 的关系 

 $2002 是 PPUSTATUS，它里边有一个非常重要的状态位： 
  - VBlank 标志 

 当 PPU 进入 VBlank 时，这个位会被置 1。 
 所以 CPU 即使不靠 NMI，也可以通过读 $2002 来知道： 

  - PPU 是不是已经进入 VBlank 了 

 但它有副作用： 
  - 读 $2002 会清掉 VBlank 标志 
  - 同时还会影响 \$2005/\$2006 的写入锁存状态 

 所以 $2002 不是“随便看看”的普通状态变量，它是会影响系统行为的。 

---------

5. 你现在终于可以理解“读状态寄存器会有副作用”这句话了 

为什么会这样设计？ 

因为在硬件协议里，“读到这个状态”本身就可能表示： 

  - “我知道了” 
  - “这次通知已经被接收了” 

 所以读 $2002 清 VBlank，你可以暂时把它理解成： 
  - CPU 读了之后，等于确认了“我已经看到这次 VBlank 状态了” 

 虽然底层机制更复杂，但这个直觉对你当前阶段够用了。 

---------

6. VBlank 和完整渲染的关系 

 你现在不需要知道整帧 262 扫描线这些细节，也能先理解 VBlank： 
  - 一帧不是瞬间完成的 
  - PPU 有“画面输出阶段” 
  - 也有“这一帧输出结束后的空档” 
  - 这个空档就是 VBlank 

 所以 VBlank 其实是“时间结构”的概念，不是某块数据区。 

---------

7. 为什么这会和你当前课程进度强相关 

 因为你前面已经学了： 
  - NMI 
  - PPUCTRL ($2000) 
  - PPUSTATUS ($2002) 
  - PPU 和 CPU 共用一条时间轴 

 现在这些终于汇合起来了： 
  - PPU 到了某个时间点进入 VBlank 
  - $2002 的 VBlank 位置 1 
  - 如果 $2000 允许，就触发 NMI 
  - CPU 进入 NMI 例程做画面更新 

 这就是 NES 游戏主循环里非常经典的节奏。 

---------

8. 你现在该如何在脑中想象一帧 

 你可以先想成这样： 

- CPU 持续跑主逻辑 
-  PPU 持续扫画面 
- 到一帧末尾 -> PPU 进入 VBlank 
- 如果允许 NMI -> 通知 CPU 
- CPU 在 NMI 里更新下一帧要用的图形状态 
- 然后回到主逻辑 


 这就是“帧同步”的雏形。 

---------

9. 当前阶段你最该掌握的不是数值，而是关系 

 先别急着背： 
  - VBlank 在第几扫描线开始 
  - 一共多少周期 

 你现在最重要的是先把关系搞清楚： 

  - VBlank 是 PPU 的状态
  - $2002 能反映这个状态
  - $2000 能决定是否把这个状态转成 NMI 通知 CPU
  - NMI 是 CPU 感知“新一帧开始准备”的关键机制

 这四者的关系一定要清楚。

---------

10. 这一课的最简总结

  - VBlank 是一帧画面结束后的空白期
  - 这是 CPU 更新 PPU 相关数据的好时机
  - $2002 里有 VBlank 状态位
  - $2000 决定是否在 VBlank 开始时触发 NMI
  - 这就是 NES 中 PPU 和 CPU 最经典的协作方式
