第 4 课：ROM 文件和 iNES 头到底是什么 这节课开始真正接近代码实现。 目标是让你搞清楚： 
  1. .nes 文件里到底装了什么 
  2. 为什么模拟器不能直接把整个文件当成“程序内存” 
  3. 什么是 iNES header 
        4. PRG、CHR、Mapper ID、Mirroring 分别是什么意思 

 这节课主要对应 doc/doc.md 的第 5 章，也和当前项目的 core/cart.c 强相关。 

---------

1. .nes 文件不是“游戏运行时内存快照” 

 这是第一件要纠正的事。 很多初学者会下意识以为： “ROM 文件是不是把 NES 运行时整块内存存进去了？” 不是。 .nes 文件更像是： 卡带内容的打包文件 它主要包含的是： 
  - 卡带的元信息 
  - CPU 要读的程序数据 
  - PPU 要读的图形数据 
  - 一些控制信息，比如 Mapper 和镜像方式 

 所以 ROM 文件不是整台 NES 的状态，只是“卡带上的内容”。 这很重要，因为模拟器启动后，还要自己创建： 
  - CPU 寄存器 
  - 主机 RAM 
  - PPU 寄存器 
  - VRAM / OAM 
  - APU 状态 
  - 输入状态 

 这些都不是从 .nes 文件直接拿来的。 

---------

2. .nes 文件的最常见结构 

 先看最重要的版本，也就是你现在先要学会的 iNES 格式。 一个 .nes 文件通常可以粗略看成：

```
  +----------------------+          
  | 16-byte iNES Header  |          
  +----------------------+         
  | Trainer (optional)   |          
  | 512 bytes            |          
  +----------------------+          
  | PRG ROM              |         
  | N * 16KB             |        
  +----------------------+          
  | CHR ROM              |         
  | M * 8KB              |       
  +----------------------+  
```

你现在先不用背细节，只记这个层次： 

  - 前 16 字节是头 
  - 头后面可能跟一个 trainer 
  - 再后面是 PRG 
  - 最后是 CHR 

 当前项目在 core/cart.c 里就是按这个顺序解析的。 

---------

3. iNES Header 是什么 

 iNES Header 就是 ROM 文件开头的 16 字节说明书。 它告诉模拟器： 
  - 这是不是真的 NES ROM 
  - PRG 有多大 
  - CHR 有多大 
  - 这张卡带用哪个 Mapper 
  - 用什么镜像方式 
  - 有没有 trainer 
  - 有没有电池 RAM 
  - 是否是 NES 2.0 扩展格式 

 你可以把它想成“卡带身份证”。 如果没有这个头，模拟器就很难知道： 
  - 该从文件哪个位置切 PRG 
  - PRG 到底有几页 
  - CHR 到底有几页 
  - 地址映射规则该选哪套 

---------

4. Header 的前 4 字节：魔数 

 最前面的 4 字节通常是： 'N' 'E' 'S' 0x1A 这就是魔数。 作用很简单： 告诉你这真的是一个 NES ROM 文件，而不是别的垃圾数据 当前项目里在 core/cart.c 里就检查了这个值。 所以第一步通常不是“开始读 PRG”，而是： 先验明正身 

5. PRG ROM 是什么 

 PRG ROM 是给 CPU 看的程序区。 你可以先把它理解成： 
  - 游戏代码 
  - 游戏常量数据 
  - CPU 执行时会读取的主要内容 

 PRG 的单位通常按 16KB 计。 比如 header 里如果说： 
  - prg_rom_pages = 2 

 那就表示： 
  - PRG ROM = 2 * 16KB = 32KB 

 这个项目也是这么做的，在 core/cart.c 里会根据页数计算偏移。 你现在先记一句： PRG 是给 CPU 读的。 

6. CHR ROM 是什么 

 CHR ROM 是给 PPU 看的图形数据区。 它一般存的是： 
  - 图块数据 
  - 精灵图案数据 
  - 背景图案数据 

 CHR 的单位通常按 8KB 计。 比如 header 里如果说： 
  - chr_rom_pages = 1 

 那就是： 
  - CHR ROM = 1 * 8KB = 8KB 

 你可以先把它想成： PRG 是程序书，CHR 是图像素材库 后面讲 PPU 时你会看到，PPU 渲染画面时会频繁去读 CHR 数据。 

7. Mapper ID 是什么 

 这个你现在先不深挖实现，只建立概念。 Mapper ID 告诉模拟器：这张卡带采用哪种地址映射规则。` 比如常见的： 
  - 0：NROM 
  - 2：UxROM 
  - 4：MMC3 

 也就是说： 
  - 即使两个 ROM 都是 .nes 
  - 它们的文件格式头部都差不多 
  - 但它们的卡带硬件规则可能完全不同 

 当前项目在 core/mapper.c 里就是根据 mapper_id 选择具体逻辑。 所以： 读 ROM 不只是读字节，还要读出“这张卡带该按什么规则解释”。 

8. Mirroring 是什么 

 这个词现在只先建立概念，不讲 PPU 细节。 Mirroring 主要和 PPU 的 name table 映射有关。 它告诉模拟器背景表该怎么镜像。 你现在先只记住它是“卡带告诉 PPU 地址空间怎么折叠/复用”的一种规则。 常见有： 
  - Horizontal 
  - Vertical 
  - Four-screen 

 当前项目也在 core/cart.c 里从 header 解析它。 你现在不需要会用，只需要知道： Mirroring 是卡带元信息的一部分，不是运行时瞎猜出来的。 

9. Trainer 和 Battery 是什么 

 这两个先只要认识。 Trainer 
  - 可选的 512 字节额外区域 
  - 不是所有 ROM 都有 
  - 有的话，PRG 起始位置会往后挪 

 Battery 
  - 表示卡带可能带电池保存 RAM 
  - 比如存档可能会用到 

 这个项目也处理了这两个信息，core/cart.c 里都能看到。 

10. 当前项目是怎么解析 ROM 的 

 这个项目的流程很典型： 
  1. 先读 header 
  2. 检查 magic 
  3. 解析是否 NES 2.0 
  4. 解析 battery / trainer / mirroring / mapper_id 
  5. 计算 PRG 和 CHR 的起始位置 
  6. 保存指向这些区域的指针 
        7. 根据 mapper_id 初始化 Mapper 

 这就是“卡带加载”的本质。 所以以后你在 Rust 里也大概率会有一个类似入口：

```rs

      pub fn from_ines(data: &[u8]) -> Result<Self, CartError> {                
          // parse header             
          // validate             
          // slice prg/chr          
          // create mapper                    
      }                              
  }  
```

11. 为什么不能直接把整个文件塞给 CPU 

 因为文件里前 16 字节是 header，不是程序。 可能还有 trainer，也不是 PRG 正文。 更重要的是： 
  - CPU 看的是 PRG 地址空间 
  - PPU 看的是 CHR 地址空间 
  - Mapper 还会改变它们的映射关系 

 所以 ROM 文件必须先“拆包”，不能直接“整体拿来执行”。 你可以把它理解成： .nes 文件是运输包装箱` 

12. 你现在应该形成的心智模型 

 看到一个 .nes 文件时，你脑子里应该自动浮现： 
  1. 先看前 16 字节 header 
  2. 判断文件是否合法 
  3. 读出 PRG 页数和 CHR 页数 
  4. 读出 Mapper ID 和 Mirroring
  5. 如果有 trainer，跳过 512 字节 
  6. 切出 PRG 数据区 
  7. 切出 CHR 数据区 
  8. 创建 Cartridge 对象 
  9. 再由 Cartridge 结合 Mapper 对外提供 CPU/PPU 读写 

 如果这条链你能顺下来，第 4 课就算懂了。 

13. 这节课最重要的 5 句话 
      
  1. .nes 文件是卡带内容的打包格式，不是整台机器的运行时快照。 
  2. iNES header 是 ROM 的说明书。 
  3. PRG 给 CPU 读，CHR 给 PPU 读。 
  4. Mapper ID 决定卡带映射规则。 
  5. 模拟器加载 ROM 的第一步不是执行，而是解析和拆分。
